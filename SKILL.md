# 项目出题循环

这套流程的目标只有一个：更快地产出贴项目、能直接复制、能持续迭代的提示词和验收结论。

默认把流程拆成三条路：

- 首轮提示词走`半严路`
- 后续修复提示词走`快路`
- 验收走`轻验收 / 标准验收 / 重验收`

主链路优先短、优先快，但不能丢这几个质量锚点：

- 先打开 Trae 项目
- 首轮要贴仓库
- 首轮要轻验重
- 验收要先看结果
- 输出要能直接复制

## 输出级硬禁令

先记最高优先级规则。只要是最终要发给用户、并且用户可能直接复制使用的内容，都必须先过这一层。

输出级内容包括：

- 生成的提示词
- `过程不满意`
- `产物不满意`
- 跟这三类内容紧挨着的简短结论补充

这些输出级内容里严禁出现：

- `稳定`
- `收口`
- `收住`
- `落下来`
- `这一轮`
- `上一轮`
- `当前轮次`
- `同上`

如果一句话必须表达“基于这次结果继续修”，也要直接改写成当前看到的问题和要补的行为，不要借助跨次承接词。

如果示例句、模板句和这条禁令冲突，一律以这条禁令为准。

## 总原则

- 只要进入验收，默认先做结果验收。
- 只有用户这次消息里真的贴了 Trae 过程信息，才叠加做过程验收。
- 如果结果没问题，但过程里明显有冗余思考、待办拆法很怪、语言跑偏、一直重复解释、执行节奏很乱，也不能直接算高质量完成。
- 如果用户没贴 Trae 过程信息，就不要额外要求过程验收；但只要结果不满意，仍然要根据结果反推一句简短的 `过程不满意`。

## 路由规则

先判断当前该走哪条路：

- 当前项目首次生成提示词：走`首轮半严路`
- 已经出过一次题，正在补问题：走`后续快路`
- 用户回来让继续、看结果、验收、生成完了：直接进`验收分级`

如果你拿不准，就按下面这条兜底：

- 新需求优先按首轮半严路
- 补问题优先按后续快路
- 看结果优先按验收分级

## 0. 先打开 Trae 项目

这一步是主链路的起点，而且必须先做，不能后移，不能懒触发，也不能省略。

无论当前是首轮提示词、后续修复提示词，还是准备进入验收，只要进入 `solo-create` 流程，默认先运行：

- `python3 /Users/fengyin/.codex/skills/solo-create/scripts/ensure_trae_project_open.py --project-path .`

执行要求：

- 这是整个流程里最先做的动作，优先级高于扫仓库、记基线、查历史、读手册、启动项目和浏览器验收
- 如果当前目录对应的 Trae 窗口已经在，它会自行跳过重复打开
- 如果没检测到当前目录对应的 Trae 窗口，就先把它打开，再继续后面的步骤
- 不要把这一步放到附录里执行，也不要只在首轮做

## 首轮半严路

首轮不走最重流程，但也不能只凭经验出题。

### 1. 必做动作

- 轻扫仓库：优先用 `rg --files`、`git status --short`、`git diff --stat`
- 看关键代码：只读和当前切入点直接相关的文件
- 判断任务类型：明确就直走，不明确才让用户选
- 轻验重：运行 `python3 /Users/fengyin/.codex/skills/solo-create/scripts/query_prompt_history.py --project-path . --task-type "<任务类型>"`
- 按需读手册：只读命中的那一本，不要把全部手册都读一遍

### 2. 基线处理

首轮默认尝试记基线，但不把它当成绝对阻塞项。

- 优先运行仓库里的 `scripts/record_round_baseline.py`
- 如果仓库里没有，再回退到 skill 自带脚本
- 如果脚本执行成功，再按标准流程继续
- 如果脚本缺失或执行失败，要明确告诉用户当前是降级处理，后续验收只能基于累计改动判断

不要因为基线失败就卡死在出题前。

### 3. 首轮去重

首轮轻验重不能砍，这是防撞题的关键。

- 重点看同前缀、同类型、同 angle 的历史
- 如果 `matched_count` 大于 0，优先换 angle，不要只改几个词
- 如果当前类型是 `缺陷修复`，首轮优先找两个彼此有关联的问题，不要总出单点小修

### 4. 首轮提示词要求

- 必须贴当前仓库
- 必须是用户可感知的问题或需求
- 默认用中文
- 不要出现文件路径、函数名、类名、实现指挥
- 不要写成任何项目都能套用的通用模板

类型判断保持这条默认规则：

- 在现有项目里新增一块用户能独立感知的新能力、新入口或新流程，默认按 `代码生成`
- 在已有能力上继续补规则、补交互、补展示，或者在原流程上往前推一点，默认按 `功能迭代`

## 后续快路

后续修复提示词默认走更短的链路，不再重复首轮那套重动作。

### 1. 必做动作

- 先看相对基线改动
- 先看上次验收确认过的问题点
- 只围绕已确认问题生成修复提示词

### 2. 默认跳过的动作

- 不再重复做历史验重
- 不再重复做任务类型选择，除非任务性质明显变了
- 不再重读整本手册，除非输出风格明显跑偏
- 不再重扫整仓库，只补读直接相关文件

### 3. 何时回到严一点的路径

命中下面任一情况，再补动作：

- 修复范围明显扩散，不再只是补问题
- 用户中途换了需求方向
- 基线缺失，累计改动已经看不清边界
- 连续两次验收都发现问题描述和实际改动对不上

### 4. 修复提示词要求

- 只写已经验出来的问题
- 不要把最初大需求整段重写一遍
- 默认写成“修复……问题……”
- 直接描述要补的行为结果，不要写过程指挥
- 发给用户前，再做一次输出级禁词检查

## 验收分级

用户在前一次提示词之后回来让继续，默认不要再聊需求，直接进入验收。

验收先判断级别，再执行，不要默认每次都跑最重的链路。

### 轻验收

满足下面任一情况，可以先走轻验收：

- 代码改动已经足够明确说明结果没做对
- 代码改动已经足够明确说明结果已落地，而且不是强交互任务
- 当前环境起不来，但代码侧证据已经足够强

轻验收默认做这些：

- 看相对基线改动
- 读真正新增或继续修改的文件
- 直接给完成判断

### 标准验收

大多数可运行任务走标准验收。

标准验收默认做这些：

- 看相对基线改动
- 启动项目
- 走一遍核心路径
- 给固定格式结论

### 重验收

命中下面任一情况，再升到重验收：

- 前端强交互任务
- 路由、滚动、回跳、多步骤表单
- 列表详情联动、筛选分页联动、跨区域联动
- 用户明确要求用 Codex Browser 实操
- 用户贴了 Trae 过程信息
- 上一次已经证明“只看代码不够，必须真点一遍”

### 浏览器规则

- 只要进入实际操作阶段，默认优先使用 Codex 内置浏览器
- 只有内置浏览器不可用、连接失败，或被能力边界客观卡住时，才允许降级
- 如果发生降级，要在验收结果里明确说明

### 过程验收触发条件

只有当用户这次消息里真的贴了 Trae 过程信息，才做过程验收。

过程验收重点按下面这些方向看，但最终只选一个最关键的问题写进 `过程不满意`：

- 有没有充分理解 prompt，还是一开始就理解偏了
- 目标是不是明确，还是做着做着方向散了
- 推理过程是不是精简，还是反复解释、反复总结、绕圈
- 输出是不是正常，比如语言跑偏、结构混乱、内容前后对不上
- 任务规划是不是合理，比如待办拆分失衡、先后顺序怪、漏关键路径
- 工具使用是不是合理高效，比如该直接做的地方来回读、反复查、工具切换过多
- 整个操作过程是不是合理，比如执行节奏拖沓、长时间停在无效分析、一直不落到动作
- 有没有高危操作，或者明显缺边界感，比如越过原有约束、擅自做危险改动
- 总结是不是正确，还是对已做内容、未做内容、结果状态说错了
- 有没有虚假完成情况，比如实际没做对，却把结果写成已经完成

如果用户没贴过程内容，就不要凭空补过程问题。

## 完成判断

完成判断收紧成三种：

- `是否完成：已完成`
- `是否完成：未完成`
- `是否完成：暂时无法判定完成`

每次给出验收结论时，都要紧跟着补一行 `修改范围：...`。

可选项只有这五个：

- `修改范围：单文件`
- `修改范围：模块内多文件`
- `修改范围：跨模块多文件`
- `修改范围：跨系统多模块`
- `修改范围：无需修改`

判断标准优先看：

- 核心行为是不是真的存在
- 有没有明显交互问题
- 有没有破坏原来的阅读流或使用流
- 如果任务和缩放、筛选、切换条目有关，顺手验证这些后续动作

不要把“代码看起来像是写了”当成完成证据。

### 修改范围判定

`修改范围` 只根据这次代码变更来判断，不根据需求大小、输出字数或主观感受来猜。

默认按下面的规则判：

- `单文件`：这次只改了一个文件
- `模块内多文件`：改了多个文件，但都还在同一个功能模块、同一个端、同一个业务区里
- `跨模块多文件`：改了多个文件，已经跨出一个模块，涉及两个及以上功能区，但还在同一个系统内
- `跨系统多模块`：同时改了不同端，或者不同系统边界里的代码，比如前端和后端、移动端和 PC 端、用户端和后台管理端
- `无需修改`：这次本来就不需要改代码，或者任务类型是 `代码理解`

这里优先记一条硬规则：

- 如果任务类型是 `代码理解`，`修改范围` 一律输出 `无需修改`

拿不准时按这条顺序收：

- 先看是否 `代码理解`
- 再看是否根本没代码改动
- 再看文件数
- 再看是否跨模块
- 最后再看是否跨系统

## 输出不满意原因

如果结果没做好，或者虽然做出来了但体验不对，就要输出不满意原因。

默认模板只有两行：

- `过程不满意：...`
- `产物不满意：...`

这两行在最终输出里只允许用一个换行隔开，不要在中间额外空一行。

### 先记四条硬约束

- `过程不满意` 默认只写一个最核心的问题点，优先压成一句话
- `过程不满意` 只描述这次暴露出来的当前问题，不要带历史背景、基线状态、工具路径或验收过程说明
- `产物不满意` 只写用户能直接看到、点到、感受到的问题现象，不要夹带代码补了什么、工具怎么验的、和前一次相比怎样
- 如果产物侧现在确实还没法直接判断，就直接写 `产物不满意：暂没法判断。`

### 写法规则

- 正文至少 30 个字
- 只写问题本身，不写客套、分析、流程说明
- 默认优先写成 1 到 3 句短句
- `过程不满意` 可以从下面这些方向里挑最贴的一个来写：理解 prompt、目标明确性、推理精简度、输出是否正常、任务规划、工具使用、操作过程、边界感、总结正确性、虚假完成
- `过程不满意` 默认不要写“已经意识到”“其实有补到”“虽然已经朝着……改了”
- `过程不满意` 默认不要写“说明这里……”“看起来像是……”“意味着……”
- `产物不满意` 默认不要写“浏览节奏”“操作节奏”“阅读节奏”
- `产物不满意` 默认不要写“代码里已经补了……”“页面看起来比前一次好一些”
- 默认不要出现工具、端口、浏览器、基线之类的背景信息
- 默认不要出现跨次对比词和输出级禁词

### 更合适的例子

- `过程不满意：规划上少了对目标切换后整套结果一致性的校验。`
- `产物不满意：目标名次和目标分数切换以后，反推出来的区间、推荐报价、预计得分和模拟名次还是各走各的，页面上看到的不是同一套结果。`

- `过程不满意：对 prompt 的理解一开始就偏了，后面一直围着错误目标在做，真正要补的联动没有落进去。`
- `产物不满意：切换目标以后，页面上真正该跟着一起变化的结果还是没变，前后显示对不上。`

- `过程不满意：实现链路上登录状态和优惠券可用数据没有处理对。`
- `产物不满意：优惠券这块现在还有问题，用户端虽然有入口，但领券页拿不到可领的券，购物车里也选不到可用券。`

- `过程不满意：推理和输出都不够精简，同一层意思反复解释，真正该确认的关键动作反而没有及时做。`
- `产物不满意：页面上该验证的关键结果还没落到实处，看到的内容和原本要补的问题没有对上。`

- `过程不满意：实现上还缺一次把预约主流程从头到尾真实走通的确认。`
- `产物不满意：暂没法判断。`

### 明确反例

- 不要写“这次看起来比前一次好一些”
- 不要写“已经意识到要补”
- 不要写“代码里也补了……”
- 不要写“内置浏览器里没点到……”
- 不要写“因为基线没补记……”
- 不要写“整体上还不能算稳定可用”
- 不要把理解偏了、目标不清、工具乱用、总结说错、虚假完成这些问题混成一长串，一次只抓最关键的一个点

## 输出修复提示词

如果验收没过，或者用户明确表示不满意，就直接生成修复提示词。

修复提示词规则：

- 必须基于已经验出来的问题来写
- 只写当前需要补的行为
- 默认写成“修复……问题……”
- 不要重写最初那份完整需求
- 不要写实现指挥、文件路径、函数名、按钮名、日志原文
- 不要夹带跨次承接词和输出级禁词

更合适的方向：

- `修复搜索完成后自动跳到第一个结果的问题，保留当前阅读位置。`
- `修复高亮范围偏移和缩放后错位的问题。`
- `修复筛选后结果变乱和分页对不上的问题，翻页后也要接着当前筛选结果显示。`

## 默认输出格式

在“前一次提示词已经跑完，用户回来让继续”的场景里，默认最终只给三类核心结果：

- `是否完成`
- `过程不满意` / `产物不满意`
- `提示词`

先记一条更高优先级的输出分流：

- 如果当前是在生成首轮提示词，最终只输出提示词正文，不要夹带任务类型、基线状态、去重说明、过程说明、额外总结或别的补充信息。
- 如果当前是在后续修复提示词或验收场景，才按下面的固定格式输出。

默认按下面的格式输出：

- `是否完成：已完成`
- `修改范围：...`

或

- `是否完成：未完成`
- `修改范围：...`
- `过程不满意：...`
- `产物不满意：...`
- `提示词：...`

或

- `是否完成：暂时无法判定完成`
- `修改范围：...`
- `过程不满意：...`
- `产物不满意：...`
- `提示词：...`

只要结果不满意，就必须同时输出 `过程不满意` 和 `产物不满意`。
最终输出时，各行之间只保留一个换行，不要在 `过程不满意` 和 `产物不满意` 中间额外插空行。

- 一个项目只对应内置浏览器里的一组固定标签页

如果两个项目端口相同，默认从原端口 `+1` 往后找可用端口，不要停已有服务。

如果两个项目依赖同一个登录态、同源浏览器存储、同一套会互相污染的数据环境，或当前没法可靠区分标签页，就不要直接并行验收。
