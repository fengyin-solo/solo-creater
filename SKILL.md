# 项目出题循环

这套流程的目标只有一个：更快地产出贴项目、能直接复制、能持续迭代的提示词和验收结论。

默认把流程拆成三条路：

- 单工程默认走`已投递验收路`
- 单工程标准完整流程走`首轮半严路`
- 后续修复提示词走`快路`
- 验收走`轻验收 / 标准验收 / 重验收`
- 父目录源仓库复制到 GitHub 多仓走`批量建仓路`
- 父目录批量出题走`批量首轮路`
- 父目录批量投递走`批量投递路`

主链路优先短、优先快，但不能丢这几个质量锚点：

- 单项目先打开 Trae 项目
- 批量建仓先确认 GitHub CLI 可用，先 dry-run，再真实创建
- 批量生成先不打开 Trae，生成后再按需打开和发送
- 首轮要贴仓库
- 首轮要轻验重
- 验收要先看结果
- 输出要能直接复制

## 输出级硬禁令

先记最高优先级规则。只要是最终要发给用户、并且用户可能直接复制使用的内容，都必须先过这一层。

输出级内容包括：

- 生成的提示词
- `过程不满意`
- `产物不满意`
- 跟这三类内容紧挨着的简短结论补充

这些输出级内容里严禁出现：

- `稳定`
- `收口`
- `收住`
- `落下来`
- `这一轮`
- `上一轮`
- `当前轮次`
- `同上`
- `（继续）`

如果一句话必须表达“基于这次结果继续修”，也要直接改写成当前看到的问题和要补的行为，不要借助跨次承接词。

这里的禁令不只是禁掉词本身，也禁掉带前缀、带括号、带引用感的写法，比如：

- `（继续）`
- `上一轮`
- `同上`
- 任何把当前问题写成“承接上文”“接着前面”的前缀式表达

如果示例句、模板句和这条禁令冲突，一律以这条禁令为准。

输出级内容还有两条格式硬规则：

- `过程不满意` 不能以页面、按钮、文案、提示或内容显示英文作为理由；英文显示属于产物现象，只能放进 `产物不满意`
- `过程不满意`、`产物不满意` 和 `提示词` 每一行在换行前都必须以标点结尾，优先用 `。`、`！`、`？`；不要让不满意原因行尾没有标点

## 总原则

- 只要进入验收，默认先做结果验收。
- 只有用户这次消息里真的贴了 Trae 过程信息，才叠加做过程验收。
- 如果结果没问题，但过程里明显有冗余思考、待办拆法很怪、语言跑偏、一直重复解释、执行节奏很乱，也不能直接算高质量完成。
- 如果用户没贴 Trae 过程信息，就不要额外要求过程验收；但只要结果不满意，仍然要根据结果反推一句简短的 `过程不满意`。

## 路由规则

先判断当前该走哪条路：

- 当前目录是单个工程时，先检查上级目录是否有 `solo-create-prompts.xlsx` 且当前文件夹有对应提示词；如果命中，默认走`已投递验收路`
- 用户明确要求“单个标准完整流程”“重新生成首轮提示词”“从头生成提示词”时，走`首轮半严路`
- 用户要求在父目录下把一个源代码仓库批量复制成多个 GitHub 仓库，或提到“建多个仓库 / 复制仓库 / GitHub 批量建仓 / 初始化多个源码一样的仓库”，走`批量建仓路`
- 用户要求选择父目录、遍历多个子文件夹、批量生成提示词或写入 Excel：走`批量首轮路`
- 用户在父目录批量提示词已生成后发送编号范围，或要求打开对应 Trae 窗口并自动发送：走`批量投递路`
- 已经出过一次题，正在补问题：走`后续快路`
- 用户回来让继续、看结果、验收、生成完了：直接进`验收分级`

如果你拿不准，就按下面这条兜底：

- 单个工程默认先尝试已投递验收路
- 只有用户明确要从头出题，才按首轮半严路
- 明确提到父目录里只有一个源仓库、要复制成多个 GitHub 仓库时优先按批量建仓路
- 明确提到父目录或子文件夹批量生成时优先按批量首轮路
- 明确提到编号范围、前几个、打开 Trae、自动发送时优先按批量投递路
- 补问题优先按后续快路
- 看结果优先按验收分级

## 单工程两类流程

单工程目录下运行 `solo-create` 时分成两类。第二类是最高频默认模式。

### 1. 标准完整流程

仅在用户明确要求从头生成首轮提示词时使用。

触发示例：

- `使用 $solo-create，按单个标准完整流程跑`
- `重新生成首轮提示词`
- `从头出题并验收`

执行方式：

- 先打开当前 Trae 项目
- 走首轮半严路生成首轮提示词
- 发送首轮提示词给用户
- 直接进入验收分级
- 如果验收未通过，走后续快路生成修复提示词

### 2. 已投递验收路

这是单工程默认模式。适用于批量流程已经生成对应提示词，并且已经打开 Trae 窗口、发送过提示词；用户手动切到某个子项目目录后执行 `solo-create`。

进入单工程目录后，先运行：

- `python3 /Users/fengyin/.codex/skills/solo-create/scripts/batch_prompt_workbook.py locate-project --project .`

如果返回 `found: true`，且记录里 `提示词` 非空：

- 不再生成首轮提示词
- 不再重复历史验重
- 不再重复读任务手册来出题
- 把 Excel 里的 `提示词` 作为本项目已经发送给 Trae 的任务依据
- 如果 `状态` 是 `已发送`，先检查用户这次消息里是否已经明确发送 `Trae Session ID`；如果没有，立即停止，并提醒用户先发送 `Trae Session ID`
- 如果 `状态` 是 `已生成` 但不是 `已发送`，先提醒这条提示词可能还没有投递；如果当前 Trae 窗口已经能确认任务完成痕迹，也仍然必须先等用户发送 `Trae Session ID` 后才能进入验收
- 验收没过时，按后续快路生成修复提示词

如果返回 `found: false`，说明当前工程不是批量 Excel 管理的子项目；除非用户另有说明，再退回标准完整流程。

## 批量建仓路

当父目录初始化时只有一个源代码仓库，而用户要求批量复制成多个 GitHub 仓库时，走这条路。

默认使用官方 GitHub CLI，因为它能处理登录身份、远端仓库创建和可见性。建仓脚本默认把远端创建和 push 分开执行：先用 `gh repo create` 创建仓库，再用 `gh auth token` 通过临时 HTTPS 凭据推送，最后把本地 `origin` 和分支 upstream 清理成不含 token 的普通 HTTPS 地址。

### 1. 触发和前置检查

触发示例：

- `把这个源仓库复制成默认 43 个 GitHub 仓库`
- `父目录下只有一个源码仓库，按 solo-create 默认分布建多个仓库`
- `用 gh 批量创建代码生成和 feature 迭代仓库`

执行前先确认：

- 当前目录或用户提供路径是父目录
- 父目录下只有一个明确的源代码仓库，或用户已经通过 `--source` 指定源仓库
- 本机安装并登录了官方 GitHub CLI，可用 `gh auth status` 检查；如果 `gh repo create --help` 看不到官方创建仓库说明，说明 `gh` 可能被其他 npm 包或工具占用，必须先换成官方 GitHub CLI。
- 如果用户指定 `--owner`，必须确认 `gh auth status` 里的 active account 有权在该 owner 下创建仓库；如果没有，先运行官方 `gh auth login --hostname github.com --git-protocol ssh --web --skip-ssh-key`，并让用户在网页登录目标账号授权。
- 如果用户要求“本地 git 账号改成某账号”，优先使用 `--git-user-name` 和 `--git-user-email` 写入每个新仓库的本地提交身份；不要只改全局配置后假设脚本会继承到已生成目录。
- 源仓库必须是干净 git 工作区；如果有未提交改动，先停下让用户处理，除非用户明确允许复制脏工作区
- 新建仓库默认必须是 `public`，其他人都能访问；只有用户明确要求私有仓库时才允许改成 `--visibility private`
- 真实建仓前必须先 dry-run，确认仓库名称、数量、owner 和本地目标目录

### 2. 默认仓库分布

默认创建 43 个仓库：

- `代码生成` 20 个
- `功能迭代` 20 个，对应命名标识 `feature`
- `代码理解` 1 个
- `代码重构` 1 个
- `工程化` 1 个

默认命名格式：

- `<源项目编号>-<任务类型标识>-<序号>`
- `序号` 必须在所有任务类型之间全局拉通递增，不允许每个任务类型从 1 重新开始。
- 默认 43 个仓库的序号范围是：`codegen` 为 1-20，`feature` 为 21-40，`understand` 为 41，`refactor` 为 42，`engineering` 为 43。

GitHub 仓库名和本地文件夹名必须使用 ASCII 任务标识，避免 GitHub 把中文仓库名规整成缺失任务类型的名字：

- `代码生成` 使用 `codegen`
- `功能迭代` 使用 `feature`
- `代码理解` 使用 `understand`
- `代码重构` 使用 `refactor`
- `工程化` 使用 `engineering`

示例：

- `02133-codegen-1`
- `02133-feature-21`
- `02133-understand-41`
- `02133-refactor-42`
- `02133-engineering-43`

源项目编号优先从源仓库目录名中提取连续数字；如果目录名里没有编号，必须让用户指定 `--source-number`，不要自己编。

### 3. 建仓脚本

先 dry-run：

- `python3 /Users/fengyin/.codex/skills/solo-create/scripts/create_batch_github_repos.py --parent "<父目录>" --source "<源仓库目录>" --dry-run`

真实创建：

- `python3 /Users/fengyin/.codex/skills/solo-create/scripts/create_batch_github_repos.py --parent "<父目录>" --source "<源仓库目录>"`

常用参数：

- `--owner "<GitHub 用户或组织>"`：指定创建到哪个用户或组织下；不填时使用 GitHub CLI 当前默认账号
- `--visibility public|private|internal`：默认 `public`，保证新建仓库开放给其他人访问
- `--source-number "<编号>"`：目录名无法提取源项目编号时显式指定
- `--codegen-count 20`
- `--feature-count 20`
- `--understand-count 1`
- `--refactor-count 1`
- `--engineering-count 1`
- `--gh-bin "<官方 gh 路径>"`：当系统里的 `gh` 被其他工具占用时，显式指定官方 GitHub CLI
- `--git-user-name "<提交用户名>"`：写入每个新仓库的本地 `user.name`
- `--git-user-email "<提交邮箱>"`：写入每个新仓库的本地 `user.email`
- `--allow-dirty`：用户明确允许复制带未提交改动的源仓库时才使用

脚本行为：

- 为每个目标仓库创建同名本地子目录
- 复制源仓库工作区内容，但不复制源仓库的 `.git`
- 在每个目标目录重新 `git init`、提交初始代码
- 使用 `gh repo create` 创建 GitHub 远端仓库
- 使用 `gh auth token` 通过临时 HTTPS 凭据 push 初始提交，避免本机 SSH key 仍指向其他账号时推送失败
- push 后把 `origin` 设置为 `https://github.com/<owner>/<repo>.git`，并把分支 upstream 设置为普通 `origin/<branch>`
- 如果本地目标目录已存在且远端也存在，脚本会尝试补推缺失的远端分支；如果远端分支已经存在，则跳过并记录原因
- 如果本地目标目录或远端仓库已存在，默认不删除、不覆盖源码内容
- 输出 JSON 汇总，包含 planned、created、skipped、failed

建仓后必须做三项核对：

- 检查 JSON 汇总里的 `failed_count` 必须是 `0`
- 抽查本地目录的 `git remote get-url origin`，必须是不含 token 的普通 `https://github.com/<owner>/<repo>.git`
- 用 `gh api repos/<owner>/<repo>/branches/<branch>` 或等价命令核对远端 `main` 分支数量；默认 43 个仓库必须全部有 `main`

如果中途失败：

- 先用 `ps` 停掉残留的 `create_batch_github_repos.py` 或 `gh repo create` 进程，避免继续推错 owner
- 如果只是 push 失败且远端已创建，不要删除远端；重新运行同一条脚本，它会对已有本地目录和远端补推缺失分支
- 如果本地目录使用了错误提交身份或错误命名，确认源仓库不在删除范围内后，删除本次新建的目标目录再重新 dry-run
- 如果发现 `.git/config` 里残留 `x-access-token` 或 `gho_`，必须立刻把 `origin` 改回普通 HTTPS，并清理 branch upstream 后再继续

### 4. 建仓后衔接批量首轮路

批量建仓成功后，父目录会得到一组能被批量首轮路识别的子文件夹。

下一步默认运行：

- `python3 /Users/fengyin/.codex/skills/solo-create/scripts/batch_prompt_workbook.py scan --parent "<父目录>"`

然后按`批量首轮路`继续生成提示词。

## 批量首轮路

当用户要求对一个父目录下的多个子项目批量生成提示词时，优先让用户选择或提供父目录；如果当前工作目录明显就是父目录，可以直接使用当前目录。

如果父目录下还只有一个源仓库，且用户要先得到多个源码相同的 GitHub 仓库，不要直接进入批量首轮路，先走`批量建仓路`。

### 1. 子文件夹识别

只处理名称能解析出任务类型和编号的一级子文件夹。

默认文件夹命名格式：

- `label-02133-feature迭代-22`

解析规则：

- 最后一段数字是编号，例如 `22`
- 编号前面任意一段包含任务类型，例如 `feature迭代`
- `feature`、`功能` 归一为 `功能迭代`
- `bug`、`缺陷` 归一为 `缺陷修复`
- `codegen`、`生成` 归一为 `代码生成`
- `理解` 归一为 `代码理解`
- `重构` 归一为 `代码重构`
- `工程` 归一为 `工程化`
- 无法解析任务类型或编号的子文件夹跳过，不要强行猜

### 2. Excel 断点续跑

父目录下统一维护 `solo-create-prompts.xlsx`。

进入批量流程后，先运行：

- `python3 /Users/fengyin/.codex/skills/solo-create/scripts/batch_prompt_workbook.py scan --parent "<父目录>"`

执行要求：

- 如果 Excel 不存在，脚本会创建
- 如果 Excel 已存在，读取其中已有记录
- `提示词` 列非空的子文件夹视为已生成，跳过
- 只对脚本返回的 `pending` 子文件夹继续生成
- 每完成一个子文件夹，立刻写回 Excel，不要等全部完成后再统一写

写回命令：

- `python3 /Users/fengyin/.codex/skills/solo-create/scripts/batch_prompt_workbook.py update --parent "<父目录>" --folder "<子文件夹名称>" --prompt "<最终提示词>"`

Excel 固定列：

- `子文件夹名称`
- `任务类型`
- `编号`
- `提示词`
- `状态`
- `备注`
- `更新时间`

### 3. 批量生成方式

批量生成仍然沿用`首轮半严路`的生成规则，但把工作目录切到每个子文件夹。

对每个 pending 子文件夹依次执行：

- 批量生成阶段不要打开 Trae 项目
- 轻扫仓库：在该子文件夹内运行 `rg --files`、`git status --short`、`git diff --stat`
- 只读和当前切入点直接相关的文件
- 使用从文件夹名称解析出的任务类型，不再让用户选择任务类型
- 运行历史验重：`python3 /Users/fengyin/.codex/skills/solo-create/scripts/query_prompt_history.py --project-path "<子文件夹路径>" --task-type "<任务类型>"`
- 按任务类型只读命中的一本手册
- 生成符合`首轮提示词要求`的中文提示词
- 运行输出级禁词检查
- 记录提示词历史
- 写回父目录 Excel

批量模式下的提示词去重有两层：

- 先按单项目历史去重：如果历史里同类型、同 angle 已存在，优先换 angle
- 再按本次 Excel 里已生成的提示词去重：不要输出与已生成提示词目标、角度或用户可感知结果高度相同的内容

批量模式再补一层防撞规则，专门拦截“模板骨架换皮”和“同义改写抄袭”：

- 不要把同一条提示词骨架在多个项目里反复套用，只替换业务名词、对象名或页面名
- 不要沿用高频公共短语块去拼装提示词；如果一句提示词能明显拆出多段可复用骨架短语，默认重写
- 不要把别的提示词的大意原样保留后只做同义词替换；如果改完以后主句结构、诉求顺序和动作包装几乎没变，仍然算重复
- 发现和已有提示词在目标、结构、动作顺序上过近时，不要只换几个词；要直接换切入点、换用户场景、换主诉求组织方式

### 4. 批量输出给用户

批量生成完成后，默认继续投递前 6 个已生成提示词；用户可以明确要求改成其他数量，例如“生成后只跑前 3 个”。

默认投递命令：

- `python3 /Users/fengyin/.codex/skills/solo-create/scripts/send_batch_prompts_to_trae.py --parent "<父目录>" --limit 6`

批量流程的最终回复只简要说明 Excel 位置、生成数量、跳过数量、自动发送数量和未处理原因；不要把所有提示词正文铺在聊天里。

如果部分子文件夹失败，继续处理后面的子文件夹，最后列出失败的子文件夹名称和一句失败原因。

## 批量投递路

当父目录下已经存在 `solo-create-prompts.xlsx`，且用户发送编号范围、要求打开对应 Trae 窗口、粘贴提示词并自动发送时，走批量投递路。

触发示例：

- `22-27`
- `发送 22 到 27`
- `打开 22-27 对应的 Trae 并发送`
- `跑前 6 个`

执行规则：

- 先确认当前目录或用户提供路径是父目录

## 首轮半严路

首轮不走最重流程，但也不能只凭经验出题。

### 1. 必做动作

- 轻扫仓库：优先用 `rg --files`、`git status --short`、`git diff --stat`
- 看关键代码：只读和当前切入点直接相关的文件
- 判断任务类型：明确就直走，不明确才让用户选
- 轻验重：运行 `python3 /Users/fengyin/.codex/skills/solo-create/scripts/query_prompt_history.py --project-path . --task-type "<任务类型>"`
- 按需读手册：只读命中的那一本，不要把全部手册都读一遍

### 2. 基线处理

首轮默认尝试记基线，但不把它当成绝对阻塞项。

- 优先运行仓库里的 `scripts/record_round_baseline.py`
- 如果仓库里没有，再回退到 skill 自带脚本
- 如果脚本执行成功，再按标准流程继续
- 如果脚本缺失或执行失败，要明确告诉用户当前是降级处理，后续验收只能基于累计改动判断

不要因为基线失败就卡死在出题前。

### 3. 首轮去重

首轮轻验重不能砍，这是防撞题的关键。

- 重点看同前缀、同类型、同 angle 的历史
- 如果 `matched_count` 大于 0，优先换 angle，不要只改几个词
- 如果当前类型是 `缺陷修复`，首轮优先找两个彼此有关联的问题，不要总出单点小修
- 不要沿用“新增xxx功能，支持……并……”这类高频骨架反复换皮；如果同一类项目已经多次出现相近句式，优先改写成不同的用户任务入口、结果承诺或业务场景
- 不要把历史提示词的动作链顺序原样保留后只替换对象词；发现主句结构接近时，直接换整句组织方式，不要做同义词替换

### 4. 首轮提示词要求

- 必须贴当前仓库
- 必须是用户可感知的问题或需求
- 默认用中文
- 不要出现文件路径、函数名、类名、实现指挥
- 不要写成任何项目都能套用的通用模板
- 如果任务类型是 `代码生成`，提示词默认直接写“要新增的能力本身”，不要先交代“目前哪个模块只有什么功能”“现在已有哪几个入口”“现有页面暂时只能做什么”
- `代码生成` 提示词默认从新增能力起笔，优先写成 `新增xxx功能，……` 这种句式；只有少数天然不适合带“功能”二字的表达，才退回 `新增……`、`提供……`、`支持……`
- 如果新增的是一个明确的页面、入口、流程、专区、中心、能力或模块，默认补全成 `新增xxx功能`，不要只写 `活动报名`、`资料下载中心`、`订阅中心` 这类裸能力名
- 不要用“现在只有……，希望新增……”这种把现状铺在前面的句式
- 只在不补一句背景就无法理解新增能力时，才允许补极短业务上下文；这句背景也不能写成对现有功能清单的复述
- 如果一句提示词删掉“目前 / 现在 / 已有 / 现有 / 原来”这类现状前缀后仍然成立，默认删掉再输出

类型判断保持这条默认规则：

- 在现有项目里新增一块用户能独立感知的新能力、新入口或新流程，默认按 `代码生成`
- 在已有能力上继续补规则、补交互、补展示，或者在原流程上往前推一点，默认按 `功能迭代`

类型判断再补一条口径，专门防止 `代码生成` 被写成 `功能迭代`：

- 如果核心目标是“做出一个原来没有的独立能力 / 入口 / 流程”，即使它挂在现有页面里，也优先按 `代码生成`
- 如果核心目标只是“把现有能力补完整、补联动、补展示、补规则”，才按 `功能迭代`
- 不要因为题面里提到了某个现有页面、现有模块或现有入口，就把本该是 `代码生成` 的题写成 `功能迭代`

### 5. 单项目首次运行后的自动推进

如果当前是单个工程目录首次运行 `solo-create`：

- 先按首轮半严路生成首轮提示词
- 把首轮提示词发送给用户后，直接进入验收环节
- 验收环节按已有验收分级执行
- 如果验收未通过，再按后续快路生成修复提示词
- 后续流程继续沿用单项目流程

这里的“直接进入验收”指同一次执行里继续做结果检查，不要只停在首轮提示词输出。

## 后续快路

后续修复提示词默认走更短的链路，不再重复首轮那套重动作。

### 1. 必做动作

- 先看相对基线改动
- 先看上次验收确认过的问题点
- 只围绕已确认问题生成修复提示词

### 2. 默认跳过的动作

- 不再重复做历史验重
- 不再重复做任务类型选择，除非任务性质明显变了
- 不再重读整本手册，除非输出风格明显跑偏
- 不再重扫整仓库，只补读直接相关文件

### 3. 何时回到严一点的路径

命中下面任一情况，再补动作：

- 修复范围明显扩散，不再只是补问题
- 用户中途换了需求方向
- 基线缺失，累计改动已经看不清边界
- 连续两次验收都发现问题描述和实际改动对不上

### 4. 修复提示词要求

- 只写已经验出来的问题
- 不要把最初大需求整段重写一遍
- 默认写成“修复……问题……”
- 直接描述要补的行为结果，不要写过程指挥
- 发给用户前，再做一次输出级禁词检查
- 不要和这次 `过程不满意` 形成近似复述；`过程不满意` 负责解释模型在哪一步做错了什么，`下一轮提示词` 只负责明确下一步要补的用户结果
- 不要和这次 `产物不满意` 形成近似复述；`产物不满意` 负责描述现象和证据，`下一轮提示词` 必须转成新的修复目标句，不要把现象原句搬过去
- 不要和本项目历史修复提示词只做同义词替换；如果核心动作、主语结构和结果承诺没变，必须重写
- 如果存在多个待修点，不要连续输出同一骨架的并列句；优先合并成一个主问题，或换成新的主链路表达
- 写完后至少自查三组关系：`下一轮提示词` 和 `过程不满意` 是否雷同、`下一轮提示词` 和 `产物不满意` 是否雷同、`下一轮提示词` 和历史修复提示词是否只是换词

## 验收分级

用户在前一次提示词之后回来让继续，默认不要再聊需求，直接进入验收。

验收先看代码变更，再判断级别。不要默认每次都启动项目或浏览器。

进入验收时先补一条强制动作：

- `Trae Session ID` 必须由用户在本次对话里主动发送；不要自己去 Trae 窗口、代码仓库、提交记录、界面文本或别的地方代找
- 只要需要审核，就必须先收到用户发送的 `Trae Session ID`
- 这个 `Trae Session ID` 是当前验收的唯一提交标识，后面提交和 push 都要复用它
- 如果用户这次消息里还没有发 `Trae Session ID`，立即停止，不做代码验收、不做浏览器验收、不做提交检查，也不要给验收结论
- 停止时只提醒用户发送 `Trae Session ID`，不要夹带验收判断、问题分析或猜测性结论

最高优先级判断：

- 如果通过代码变更已经能判断问题是否完成，就直接给结论，不用启动浏览器验证
- 如果通过代码变更已经能判断结果没做对，也不用启动浏览器确认
- 如果是纯后端项目，不启动浏览器；优先通过代码、接口逻辑、数据流、测试或命令行结果判断
- 只有代码证据不够、且任务确实依赖前端交互或页面表现时，才进入浏览器验证

### 轻验收

满足下面任一情况，可以先走轻验收：

- 代码改动已经足够明确说明结果没做对
- 代码改动已经足够明确说明结果已落地，而且不是强交互任务
- 代码变更已经覆盖任务关键路径，继续启动浏览器不会增加判断依据
- 项目是纯后端、脚本、CLI、服务端接口或数据处理任务
- 当前环境起不来，但代码侧证据已经足够强

轻验收默认做这些：

- 看相对基线改动
- 读真正新增或继续修改的文件
- 按需要运行后端测试、类型检查、接口相关命令或最小命令行验证
- 直接给完成判断

### 标准验收

只有代码侧证据不够、但本地可运行且需要结果确认时，才走标准验收。

标准验收默认做这些：

- 看相对基线改动
- 判断项目类型：前端任务才考虑启动页面；纯后端不走浏览器
- 启动项目或运行必要命令
- 走一遍核心路径或后端最小验证路径
- 给固定格式结论

### 重验收

命中下面任一情况，再升到重验收：

- 前端强交互任务
- 路由、滚动、回跳、多步骤表单
- 列表详情联动、筛选分页联动、跨区域联动
- 用户明确要求用 Codex Browser 实操
- 用户贴了 Trae 过程信息
- 上一次已经证明“只看代码不够，必须真点一遍”

### 浏览器规则

- 浏览器不是默认验收动作；只有代码变更无法判断、且任务涉及前端页面或交互时才使用
- 纯后端项目、CLI 项目、数据处理项目、脚本项目不启动浏览器
- 如果通过代码变更已经能判断完成或未完成，不要为了补流程再打开浏览器
- 需要浏览器时，默认优先使用 Codex 内置浏览器
- 只有内置浏览器不可用、连接失败，或被能力边界客观卡住时，才允许降级
- 如果发生降级，要在验收结果里明确说明

### 验收闭环提交规则

- 只要进入验收并且代码是围绕当前提示词改过的，就必须检查当前分支是否已经提交到本地并 push 到远程
- 只要进入验收且工作区里存在围绕当前提示词的改动，不管结果最终判成 `已完成`、`未完成` 还是 `暂时无法判定完成`，都必须先用 `Trae Session ID` 提交一次，再尝试 push 到远程，然后再输出验收结论
- 验收通过前，必须确认针对当前提示词修改后的代码已经 push 到远程；不要只停在本地提交
- 用于 push 的提交 message 必须严格等于 `Trae Session ID`，不要追加前后缀、解释文字或别的编号
- 如果当前分支上还没有以该 `Trae Session ID` 为 message 的提交，先补提交再 push，再继续给最终验收结论
- 如果当前分支上已经有一个 message 精确等于该 `Trae Session ID` 的提交，但工作区里还有围绕当前提示词的新改动，仍然要把这些新改动整理进一次新的提交流程；默认做法是先确认是否允许 amend，若用户没有明确允许 amend，就继续补充代码后再用新的验收时点重新处理，不要跳过“提交并 push”动作
- 如果已经有多个提交，但没有一个提交 message 精确等于该 `Trae Session ID`，仍然不算满足要求
- 验收结果里的 `Repo URL` 必须取当前仓库远端 `origin` 对应的 GitHub 仓库地址，并且只输出最终 URL 本体，例如 `https://github.com/owner/repo`；不要手写猜测 owner 或 repo，不要带 `[owner/repo]` 这段文本
- 验收完成后最终返回的 `Commit ID`，默认填这个已 push 到远端的完整 40 位提交哈希；不要填短 hash，不要自己截断，不要填工作区状态、分支名或别的提交
- 验收结果里的 `Trae Session ID` 必须原样回填用户本次对话里发送的那条值，不要改写，不要截断，不要追加说明
- 如果因为远端不可达、权限不足或仓库未配置 upstream 导致没法 push，就按 `暂时无法判定完成` 处理，并在 `过程不满意` 或 `产物不满意` 里明确指出阻塞点

### 过程验收触发条件

只有当用户这次消息里真的贴了 Trae 过程信息，才做过程验收。

过程验收重点按下面这些方向看，但最终只选一个最关键的问题写进 `过程不满意`：

- 有没有充分理解 prompt，还是一开始就理解偏了
- 目标是不是明确，还是做着做着方向散了
- 推理过程是不是精简，还是反复解释、反复总结、绕圈
- 输出是不是正常，比如语言跑偏、结构混乱、内容前后对不上
- 任务规划是不是合理，比如待办拆分失衡、先后顺序怪、漏关键路径
- 工具使用是不是合理高效，比如该直接做的地方来回读、反复查、工具切换过多
- 整个操作过程是不是合理，比如执行节奏拖沓、长时间停在无效分析、一直不落到动作
- 有没有高危操作，或者明显缺边界感，比如越过原有约束、擅自做危险改动
- 总结是不是正确，还是对已做内容、未做内容、结果状态说错了
- 有没有虚假完成情况，比如实际没做对，却把结果写成已经完成

如果用户没贴过程内容，就不要凭空补过程问题。
但只要结果不满意，仍然要从当前结果里反推一句合规的 `过程不满意`，写法只能走下面两条之一：

- 写成对模型行为的评价：例如理解偏了、目标拆分偏了、关键主路径漏了、验证顺序不对、结果判断过早
- 写成导致结果失败的代码级或业务逻辑级根因：例如字段没有进入同一条数据链、状态机分支断开、权限规则拆成了两套、映射约束从一开始就没建好

这里有一条硬限制：

- 不要把业务 Bug 直接换个壳写成 `实现上少了校验`、`没有核对结果`、`没有处理好`、`没保住状态` 这类空话；如果不是在评价模型行为，就必须明确指出更底层的根因是什么
- 不要只写“某状态没更新、某读取先命中、某数据链断开”这类代码逻辑现象；这会被判成代码逻辑桶，不算过程与深度评价
- 如果写模型行为评价，要点名思考、规划、任务拆分、主链路识别、验证顺序或完成判断里的一个问题
- 如果写代码级 / 业务逻辑级根因，必须比 bug 现象更深一层，落到状态归属、数据源口径、写读模型、派生指标消费链、唯一记录约束、业务对象生命周期这类根因

这里再单独补一条打回高频规则，优先级和上面一样高：

- `规则 15：过程与深度评价缺失`。如果 `过程不满意` 本质上只是在描述业务 Bug 现象、代码没更新、数据没读到、状态没同步，这种写法一律视为 `代码逻辑桶`
- 遇到这种情况，必须二选一重写：要么改成模型行为评价，要么继续往下追一层，改成代码级 / 业务逻辑级根因
- 模型行为评价重点写：理解偏了、主链路抓偏了、任务拆分漏关键联动、验证顺序不对、过早下完成判断
- 根因重点写：为什么会改出这个失败结构，例如结果归属口径没统一、业务对象边界没先定义、写读模型冲突、派生指标消费链没建好、唯一记录约束缺失
- 改写时先自查一句：这句话是在点评模型怎么做错了，还是在解释为什么会形成这个错误结构；如果两者都不是，多半还在代码逻辑桶里

## 完成判断

完成判断收紧成三种：

- `是否完成：已完成`
- `是否完成：未完成`
- `是否完成：暂时无法判定完成`

判断标准优先看：

- 核心行为是不是真的存在
- 有没有明显交互问题
- 有没有破坏原来的阅读流或使用流
- 如果任务和缩放、筛选、切换条目有关，顺手验证这些后续动作

不要把“代码看起来像是写了”当成完成证据。
也不要把“本地已经提交但还没 push”当成完成证据。

完成状态再补两条硬规则：

- 只有结果满足验收要求，且对应代码已经以 `Trae Session ID` 为 message 提交并 push 到远程，才允许输出 `是否完成：已完成`
- 如果结果看起来已经做对，但还没拿到 `Trae Session ID`、还没按该 ID 提交，或还没 push 到远程，都不能输出 `已完成`

## 输出不满意原因

如果结果没做好，或者虽然做出来了但体验不对，就要输出不满意原因。

默认模板只有两行：

- `过程不满意：...`
- `产物不满意：...`

这两行在最终输出里只允许用一个换行隔开，不要在中间额外空一行。

### 先记四条硬约束

- `过程不满意` 默认只写一个最核心的问题点，优先压成一句话
- `过程不满意` 只描述这次暴露出来的当前问题，不要带历史背景、基线状态、工具路径或验收过程说明
- `产物不满意` 只写用户能直接看到、点到、感受到的问题现象，不要夹带代码补了什么、工具怎么验的、和前一次相比怎样
- 如果产物侧现在确实还没法直接判断，就直接写 `产物不满意：暂没法判断。`
- `过程不满意`、`产物不满意` 两行都必须以标点结尾，不允许行尾裸句换行

### 三要素硬要求

一条合格的不满意原因必须同时具备这三要素，缺一不可：

- `触发节点（When）`：在哪个阶段、哪一步或哪个动作出了问题
- `实际行为（What）`：模型做错了什么具体行为
- `业务影响（Impact）`：这个错误最终导致了什么后果

一条更好的不满意原因，除了三要素，还要尽量继续补这两点：

- `根因`：为什么会出现这个错误行为
- `正确做法`：当时应该怎么做才对

如果一条数据里存在多个问题点，每一个问题点都要各自满足三要素；写作格式可以不完全一致，但三要素本身是硬要求。

除了三要素，还要过一层“去雷同”检查：

- `过程不满意` 和 `产物不满意` 不能只是在同一现象上换一种说法
- `过程不满意` 要落在模型行为或更深一层根因，`产物不满意` 要落在用户可见现象和需求缺口，两行分工必须明显不同
- 当前项目本次输出里的 `过程不满意` 不能和历史 `过程不满意` 只做节点词或业务名词替换
- 当前项目本次输出里的 `产物不满意` 不能和历史 `产物不满意` 只做功能名或页面名替换
- 如果两行共享了大段主干结构，优先重写 `过程不满意` 的行为链或 `产物不满意` 的用户现象链，不要只改个别词

过程问题默认参考这个写法：

- `在【第几步/哪个环节】，模型【做了什么具体行为】→ 导致【什么后果】（→ 【根因和正确做法】）。`

产物问题默认参考这个写法：

- `产物在【哪个文件/功能】存在【什么具体问题】，表现为【客观证据，如报错/缺失需求】，无法满足【哪条需求】。`

如果产物问题本身就是由过程问题直接导致的，优先写成更完整的闭环句：

- `在【第几步/哪个环节】，模型【做了什么具体行为】→ 导致产物在【哪个文件/功能】存在【什么具体问题】，表现为【客观证据，如报错/缺失需求】，无法满足【哪条需求】（→ 【根因和正确做法】）。`

### 写法规则

- 正文至少 30 个字
- 只写问题本身，不写客套、分析、流程说明
- 默认优先写成 1 到 3 句短句
- 无论是 `过程不满意` 还是 `产物不满意`，都必须把 `触发节点`、`实际行为`、`业务影响` 三要素写出来；不要只留结论，不要只写现象
- `过程不满意`、`产物不满意`、`下一轮提示词` 这三项必须分别承担不同角色，不允许两两互相改写后复用
- `过程不满意` 不要照着 `产物不满意` 的现象句回写成“因为没更新所以没显示”这类平移句
- `产物不满意` 不要照着 `过程不满意` 的行为句回写成“因为没验到所以功能没好”这类因果复述句
- 如果一眼能看出两句只是共享同一条句法骨架，换了少量业务词、页面词或动词，就必须重写
- 如果用户没贴 Trae 过程，`过程不满意` 默认优先写“模型行为评价”；只有能说清更底层根因时，才写“代码级 / 业务逻辑级根因”
- `过程不满意` 可以从下面这些方向里挑最贴的一个来写：理解 prompt、目标明确性、推理精简度、输出是否正常、任务规划、工具使用、操作过程、边界感、总结正确性、虚假完成
- 如果写模型行为评价，优先落到这些词上：理解偏了、目标抓偏了、任务拆分漏主链路、验证顺序不对、过早下完成判断
- 如果写根因，优先落到这些词上：数据源口径不统一、派生指标消费链没建立、写读模型冲突、业务对象生命周期拆裂、唯一记录约束缺失、结果归属口径不统一
- `过程不满意` 默认要把“哪一步出了问题、模型做错了什么、造成了什么后果”写完整；如果还能继续指出根因和正确做法，优先补上
- `产物不满意` 默认要把“哪个文件或功能出了问题、具体问题是什么、客观证据是什么、没满足哪条需求”写完整；不要只写一句笼统现象
- 不要把代码现象当根因：`状态更新链路没有重算`、`读取先命中空计划`、`新增数据没展示` 这类句子要继续往上追问为什么会这样
- 根因句要能回答“为什么模型会改出这个失败结构”，而不只是复述“哪里没更新、哪里没读到、哪里断开了”
- 如果一句 `过程不满意` 可以直接替换成“某 Bug 还在”“某值没更新”“某数据没显示”，说明深度还不够，必须继续改写
- 如果产物问题已经写了用户现象，`过程不满意` 就不要再平移复述同一现象，必须补模型行为判断或更底层根因
- `过程不满意` 默认不要写“已经意识到”“其实有补到”“虽然已经朝着……改了”
- `过程不满意` 默认不要写“说明这里……”“看起来像是……”“意味着……”
- `过程不满意` 默认不要写“少了校验”“没有核对”“没有处理好”“没有保住”“没有校准完整”这种没有落到模型行为或底层根因的空壳句
- `过程不满意` 不要写“页面显示英文”“按钮文案是英文”“提示语没有中文化”这类理由；如果英文文案影响用户体验，只能写进 `产物不满意`
- `产物不满意` 默认不要写“浏览节奏”“操作节奏”“阅读节奏”
- `产物不满意` 默认不要写“代码里已经补了……”“页面看起来比前一次好一些”
- 默认不要出现工具、端口、浏览器、基线之类的背景信息
- 默认不要出现跨次对比词和输出级禁词
- 默认不要在 `过程不满意` 或 `产物不满意` 开头套 `（继续）`、`上一轮`、`同上` 这类前缀或引用式承接
- 如果当前打回备注明确指出任务类型不匹配，先改正任务类型，再继续生成不满意原因，不要带着错误任务类型继续写
- 如果当前打回备注明确指出命中“模板骨架雷同”或“同义改写抄袭”，不要在原句上修补；直接整句推倒重写，换主句结构、换问题落点、换结果表达方式

### 更合适的例子

- `过程不满意：在结果验收环节，模型只检查了打卡动作有没有触发，没有把统计卡片和连续天数当成同一条主链路继续核对，导致打卡完成后的即时反馈问题被漏掉；根因是验收范围只盯住单点动作，正确做法是把完成动作后的关键联动结果一起验透。`
- `产物不满意：产物在习惯打卡功能存在统计联动缺失，表现为用户点完成以后今日完成、当前连续、最长连续和本周完成率还停在旧值，无法满足“打卡后当前页面即时反馈结果”的需求。`

- `过程不满意：在数据建模环节，模型把当天日期计划拆成了空计划和带时段计划两套来源，导致后续读取口径从一开始就不一致；根因是业务对象边界没有先统一，正确做法是先锁定当天计划的唯一真实来源，再继续补录时段数据。`
- `产物不满意：产物在专注计划功能存在当天计划读取错误，表现为用户创建第一个专注时段后页面仍显示暂无专注计划，刚填写的时间安排没有出现在当天计划里，无法满足“创建后立即看到当天计划”的需求。`

- `过程不满意：在结果校验环节，模型没有把目标切换后的整套结果一致性当成必须联验的主路径，只核对了局部字段变化，导致关键联动错位没有及时发现；正确做法是把切换后的区间、报价、预计得分和模拟名次放进同一次联动校验。`
- `产物不满意：产物在目标切换功能存在联动不一致，表现为目标名次和目标分数切换以后，反推出来的区间、推荐报价、预计得分和模拟名次还是各走各的，无法满足“切换目标后整套结果保持一致”的需求。`

- `过程不满意：在理解需求环节，模型一开始就把目标抓偏了，后面持续围着错误目标推进，真正要补的联动没有进入实现范围，导致关键结果直到验收时仍然缺失；正确做法是先把 prompt 里的主链路目标确认清楚，再展开实现。`
- `产物不满意：产物在目标切换功能存在关键联动缺失，表现为切换目标以后页面上真正该跟着一起变化的结果还是没变，前后显示对不上，无法满足“切换后结果同步更新”的需求。`

- `过程不满意：在优惠券主链路实现环节，模型没有先把登录态和可用券数据源接成同一条消费链，导致后续页面虽然有入口，但可用数据没有真正流进去；正确做法是先打通登录态、领券页和购物车的同一份可用券数据口径。`
- `产物不满意：产物在优惠券功能存在可用数据缺失，表现为用户端虽然有入口，但领券页拿不到可领的券，购物车里也选不到可用券，无法满足“用户可领券并在下单时使用”的需求。`

- `过程不满意：在执行推进环节，模型把时间花在重复解释同一层意思上，没有先去做关键验证动作，导致真正该确认的结果迟迟没有落地；正确做法是压缩重复说明，优先完成会影响完成判断的核心验证。`
- `产物不满意：产物在当前验收对应功能存在结果未落地，表现为页面上该验证的关键结果还没出现，看到的内容和原本要补的问题没有对上，无法满足这次提示词要求。`

- `过程不满意：在验收环节，模型没有把预约主流程从头到尾真实走通就提前下判断，导致关键路径是否真正可用还没有被证实；正确做法是沿着预约主链路完成一次完整实走后再判断完成情况。`
- `产物不满意：暂没法判断。`

- `过程不满意：在任务拆分环节，模型把重点放在列表结构调整上，没有先锁定用户真正要用的行内编辑主路径，导致核心交互目标一开始就抓偏了；正确做法是先围绕用户直接操作的行内编辑主路径拆任务，再补外围结构调整。`
- `产物不满意：产物在报价列表功能存在行内编辑缺失，表现为报价还是只能新增后静态显示，列表里没有能直接改单位名称和报价的可用编辑框，无法满足“在列表里直接编辑报价信息”的需求。`

- `过程不满意：在数据流规划环节，模型没有把期望日期定义成后续排期和确认都要消费的核心字段，导致前后步骤虽然各自成立，却没有接成同一套预约安排；正确做法是先把期望日期定成主数据，再让后续排期和确认都消费同一字段。`
- `产物不满意：产物在预约流程功能存在日期链路断开，表现为预约页虽然能填写行业、产品和时间段，也能选顾问并提交，但前面选的期望日期没有进入后面的可预约时间和确认信息里，无法满足“前面填写的信息完整带到后续预约确认”的需求。`

### 明确反例

- 不要写“状态更新链路没有按同一份习惯记录结果统一消费，完成动作写入后，当前页面的统计卡片和连续天数没有在同一次交互里及时重算”
- 不要写“新建日期计划时把空计划和带时段的计划同时写入，读取逻辑先命中空计划，新增时段的数据链断开”
- 不要漏写触发节点、实际行为、业务影响中的任意一个
- 不要只写“某文件报错了”“某功能没做好”“某需求没满足”这种没有交代触发节点或错误行为的半句
- 不要只写“模型理解偏了”“模型没验到”“模型做错了”这种没有交代业务影响的空句
- 不要把 `过程不满意` 写成和 `产物不满意` 只有主语不同、其他结构几乎一样的两句话
- 不要把 `下一轮提示词` 直接改写自 `产物不满意`，例如只是把“无法满足……”换成“修复……问题……”
- 不要把历史提示词里的“新增 / 修复 + 能力名 + 支持 / 并 / 后可……”整段骨架继续套到新任务上
- 不要写“过程不满意：页面里还有英文文案”“过程不满意：按钮显示成英文”“过程不满意：提示内容没有翻译成中文”
- 不要写“这次看起来比前一次好一些”
- 不要写“已经意识到要补”
- 不要写“代码里也补了……”
- 不要写“内置浏览器里没点到……”
- 不要写“因为基线没补记……”
- 不要写“整体上还不能算稳定可用”
- 不要写“（继续）过程不满意：……”“上一轮过程不满意：……”“同上，产物不满意：……”
- 不要写“实现上少了校验”“没有核对结果”“没有处理好”“没有保住状态”这种把业务问题硬套成过程评价的话
- 不要把理解偏了、目标不清、工具乱用、总结说错、虚假完成这些问题混成一长串，一次只抓最关键的一个点

## 输出修复提示词

如果验收没过，或者用户明确表示不满意，就直接生成修复提示词。

修复提示词规则：

- 必须基于已经验出来的问题来写
- 只写当前需要补的行为
- 默认写成“修复……问题……”
- 不要重写最初那份完整需求
- 不要写实现指挥、文件路径、函数名、按钮名、日志原文
- 不要夹带跨次承接词和输出级禁词

再补一组强制去重规则，避免命中打回里的规则 28 和规则 38：

- 不要套用固定模板骨架后只替换业务词，尤其不要重复“修复xxx问题，支持 / 保证 / 验证……”这一类高频拼装句
- 不要把 `过程不满意` 或 `产物不满意` 的大意压缩后直接当成 `下一轮提示词`
- 不要把上一条 `下一轮提示词` 的句法结构原样保留，只换同义词、页面名、对象名或结果词
- 如果发现新提示词和已有提示词共享多段连续短语，或者 trigram 层面明显像同义改写，直接推倒重写，不要做局部修词
- 改写优先级依次是：先换用户场景，再换主链路动作，再换结果承诺，最后才是换局部措辞
- 修复提示词默认只保留一个主动作链，不要在一句里堆叠多个“并 / 以及 / 同时 / 后还要”去拼出模板感
- 写完后至少自查四组关系：和 `过程不满意` 是否雷同，和 `产物不满意` 是否雷同，和上一条 `下一轮提示词` 是否雷同，和同批次其他提示词是否共享大段骨架

更合适的方向：

- `修复搜索完成后自动跳到第一个结果的问题，保留当前阅读位置。`
- `修复高亮范围偏移和缩放后错位的问题。`
- `修复筛选后结果变乱和分页对不上的问题，翻页后也要接着当前筛选结果显示。`

首轮 `代码生成` 提示词再补两个定向例子：

- 不要写：`美食详情现在只有购买入口，希望新增到店用餐拼桌预约，用户能选用餐时段、人数并收到排队提醒。`
- 改成：`新增到店用餐拼桌预约功能，用户能选用餐时段、人数并收到排队提醒。`

- 不要写：`个人主页目前只有作品展示，希望增加约拍报名，用户能选档期、提交需求并收到确认通知。`
- 改成：`新增约拍报名功能，用户能选档期、提交需求并收到确认通知。`

- 不要写：`活动报名，用户能查看线下沙龙安排并预约参加。`
- 改成：`新增活动报名功能，用户能查看线下沙龙安排并预约参加。`

如果用户明确提到打回规则，或当前已经出现“模板骨架雷同 / 同义改写抄袭”风险，默认按下面的改写原则处理：

- 先改句子骨架，再改措辞；不要在原句上做增删词
- 先换成更贴业务主链路的表达，再决定是否保留“新增”或“修复”开头
- 先写用户要完成的动作和结果，不要先写技术包装语或验证包装语
- 一句提示词里尽量只保留一个最关键的用户目标，减少模板短语累加覆盖

- code block 里的 value 直接写正文，不要再用 `<>` 包裹 URL、提交哈希、Session ID、提示词或结论
