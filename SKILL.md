# 项目出题循环

这套流程的目标只有一个：更快地产出贴项目、能直接复制、能持续迭代的提示词和验收结论。

本 skill 必须按“先定路由，再过门禁，再执行单一路径，再按唯一输出模板返回”的顺序运行。不要边做边换路，不要把说明性文字当成可选建议。

其中：

- `$solo-create` 仍然是用户唯一主入口。
- 命中 `批量建仓路` 时，必须调用平级 skill `$solo-batch-repo-seed`。
- 命中 `批量首轮路` 时，必须调用平级 skill `$solo-batch-first-prompts`。
- 验收或修复验收产出最终验收模板时，必须调用平级 skill `$solo-acceptance-results` 回填独立结果 Excel。
- 用户即使最终只使用 `$solo-create`，这些子 skill 也只作为内部流程下沉，不改变主入口。

## 0. 执行契约

### 0.1 强制状态机

每次运行都必须先选且只选一个 route。合法 route 只有这 6 个：

- `已投递验收路`
- `首轮半严路`
- `后续快路`
- `批量建仓路`
- `批量首轮路`
- `批量投递路`

没有选出 route 之前：

- 不要扫仓库。
- 不要打开 Trae。
- 不要读手册。
- 不要启动服务。
- 不要生成提示词。
- 不要给验收结论。

### 0.2 执行前自检

进入动作前，先在内部完成这 7 项自检：

- `route` 是什么。
- 当前是否是验收场景。
- 当前是单项目还是父目录批量场景。
- 当前 `browser_policy` 是默认降级到 `@chrome`/Safari，还是强制 `@浏览器`。
- 当前是否明确指定了无头浏览器模式（`acceptance_browser_mode=headless`），还是沿用默认可见浏览器模式。
- 当前代码侧证据是否已经足以直接判 `未完成`，从而必须命中 `2.10`。
- 当前是否已经收到本次对话里的合法 `Trae Session ID`，以及这次验收要走“带 Session 提交态”还是“无 Session 只验收态”。
- 当前最终允许输出哪些字段。

如果其中任一项无法确定，优先按本文件的 `Hard Stops` 执行；仍然无法确定时，只允许向用户提出一个最小澄清问题。

### 0.3 单项目起始动作

只要 route 是单项目流程，就必须先运行：

- `python3 /Users/fengyin/.codex/skills/solo-create/scripts/ensure_trae_project_open.py --project-path .`

这一步优先级高于扫仓库、记基线、查历史、读手册、启动项目和浏览器验收。不要后移，不要懒触发，不要省略。

批量首轮路不执行这一步。批量投递路只在真正投递某个子项目时执行这一步。

### 0.4 核心数据格式

除了流程顺序，核心数据也必须先匹配 [data-contract.md](/Users/fengyin/.codex/skills/solo-create/references/data-contract.md) 里的 schema，再进入 route 判断或最终输出。

必须使用并遵守这 7 组数据对象：

- `run_context`
- `project_lookup`
- `workbook_row`
- `acceptance_context`
- `dissatisfaction_block`
- `final_output`
- `acceptance_result_row`

执行要求：

- 进入动作前，先在内部把当前上下文收束成 `run_context`。
- 读取 `locate-project` 结果时，先映射成 `project_lookup`。
- 读取或更新 Excel 时，逐行按 `workbook_row` 理解，不要自造列。
- 进入验收时，先把上下文收束成 `acceptance_context`，再判断 `已完成 / 未完成 / 暂时无法判定完成`。
- 生成不满意原因和下一轮提示词时，必须使用 `$solo-dissatisfaction`，并先满足 `dissatisfaction_block` 的字段和角色分工。
- `$solo-dissatisfaction` 可以接收包含路径、文件名、函数名、命令名等技术细节的内部证据，但返回给最终验收模板的 `过程不满意`、`产物不满意` 和 `下一轮提示词` 必须是中文自然描述，不得出现文件路径、英文单词或原始代码标识。
- 生成最终验收模板后，必须使用 `$solo-acceptance-results`，并先满足 `acceptance_result_row` 的字段和结果 Excel 回填规则。
- 最终返回给用户时，只允许输出 `final_output` 允许的形态。
- 只要涉及 `Trae Session ID`，先按 [data-contract.md](/Users/fengyin/.codex/skills/solo-create/references/data-contract.md) 里的 `0. Trae Session ID 格式` 校验；不合法就按“未收到合法 Session ID”处理。
- 只有本次验收明确走“带 Session 提交态”时，才检查 3 个环节完全一致：当前有效值、`git commit` message、最终质检结论输出值。

如果本文件与 `data-contract.md` 的字段定义冲突，一律以 `data-contract.md` 为准。

## 1. Route Selection

按下面顺序做 route 判定。命中即停，不再继续往下匹配。

### 1.1 已投递验收路

当满足下面全部条件时，必须进入 `已投递验收路`：

- 当前目录是单个工程。
- 运行：
  - `python3 /Users/fengyin/.codex/skills/solo-create/scripts/batch_prompt_workbook.py locate-project --project .`
- 返回 `found: true`。
- 记录里的 `提示词` 非空。

命中后：

- 不再生成首轮提示词。
- 不再重复历史验重。
- 不再重复读任务手册来出题。
- 把 Excel 里的 `提示词` 当成已发送给 Trae 的任务依据。

### 1.2 首轮半严路

当满足任一条件时，进入 `首轮半严路`：

- 用户明确要求“单个标准完整流程”。
- 用户明确要求“重新生成首轮提示词”。
- 用户明确要求“从头生成提示词”。
- 当前是单工程，且 `locate-project` 返回 `found: false`，且用户没有说明它属于批量 Excel 子项目。

### 1.3 批量建仓路

当用户要求在父目录下把一个源代码仓库批量复制成多个 GitHub 仓库时，进入 `批量建仓路`。典型表达包括：

- 建多个仓库。
- 复制仓库。
- GitHub 批量建仓。
- 初始化多个源码一样的仓库。

### 1.4 批量首轮路

当用户要求在父目录下遍历多个子文件夹、批量生成提示词或写入 Excel 时，进入 `批量首轮路`。

如果父目录下还只有一个源仓库，且用户的真实目标是先得到多个源码相同的 GitHub 仓库，不要进入批量首轮路，必须改走 `批量建仓路`。

### 1.5 批量投递路

当父目录下已存在 `solo-create-prompts.xlsx`，且用户给出编号范围、前几个、打开对应 Trae 窗口或要求自动发送时，进入 `批量投递路`。

### 1.6 后续快路

当已经出过一次题，用户是在补问题、修问题、继续改结果时，进入 `后续快路`。

### 1.7 验收分流

当用户回来让继续、看结果、验收、生成完了时：

- 如果已经命中 `已投递验收路`，留在该 route 内执行验收。
- 如果当前处理的是首轮提示词后的结果检查，仍然留在对应单项目 route 内执行验收。
- 不要新造第 7 条 route。验收只是 route 内的阶段，不是独立 route。

## 2. Hard Stops

下面这些门禁优先级最高。命中后必须立刻停止主流程，只允许输出指定内容。

### 2.1 验收缺少合法 Trae Session ID

这不再是默认 STOP，而是验收模式分流。

只要进入验收阶段，就先判断本次是：

- `带 Session 提交态`：用户已提供格式合法的 `Trae Session ID`，允许按本文件的提交 / push 链路执行。
- `无 Session 只验收态`：用户没有提供合法 `Trae Session ID`，或明确表示这次不提供。

进入 `无 Session 只验收态` 时必须满足：

- 继续正常做代码验收、浏览器验收和最终结论输出。
- 一定不能执行 `git commit`、`git push`、`git amend`，也不要为了凑齐字段去生成或代填 `Trae Session ID`。
- 最终验收结果里不要展示 `Trae Session ID` 和 `Commit ID`。
- 如果当前项目属于 Excel 管理子项目，验收阶段也不要再往生成提示词 Excel 写入任何验收字段；只允许通过 `$solo-acceptance-results` 回填独立结果 Excel，缺少 `Trae Session ID` 和 `Commit ID` 时对应单元格写空。

只有当用户已经提供格式合法的 `Trae Session ID` 时，才允许进入 `带 Session 提交态`。

### 2.2 已投递验收路但 Excel 状态不允许继续

如果 `locate-project` 命中后，记录状态是：

- `已发送`：可以直接进入验收；有合法 `Trae Session ID` 就走 `带 Session 提交态`，没有就走 `无 Session 只验收态`。
- `已生成` 但不是 `已发送`：必须先提醒“这条提示词可能还没有投递”；但这不再阻塞验收，仍可按当前代码和结果进入验收。

这时不要直接给验收结论。

### 2.3 批量投递时 Trae 未聚焦到目标项目

如果当前窗口标题不包含目标子文件夹名称，或无法确认聊天输入框已经聚焦：

- 记录该条失败并跳过。
- 不要更新 Excel 状态。
- 不要把提示词发到不确定输入区域。

### 2.4 无法判定 source 项目编号

批量建仓路里，如果源仓库目录名提取不出连续数字，且用户没有指定 `--source-number`：

- 立即停止。
- 只要求用户补 `--source-number`。
- 不要自己编编号。

### 2.5 路由无法确定

如果按 `Route Selection` 顺序执行后仍然无法确定 route：

- 只允许向用户提出一个最小澄清问题。
- 不要提前扫仓库或生成内容。

### 2.6 验收阶段 push / remote 阻塞

如果验收阶段出现下面任一情况：

- 远端不可达。
- 远端权限不足。
- 仓库未配置可用 upstream。
- `git push` 失败且当前无法恢复到正常推送。

本条只适用于 `带 Session 提交态`。如果当前是 `无 Session 只验收态`，根本不要进入 push / remote 流程，也不要因为 remote 状态阻塞验收输出。

必须立即停止验收结论输出。

这时只允许：

- 明确告知用户当前的 push / remote 问题是什么。
- 说明必须先恢复正常推送，之后才能继续给验收结果。

这时禁止：

- 输出 `任务完成情况`。
- 输出 `不满意原因`。
- 输出 `下一轮提示词`。
- 把 push / remote 阻塞写进 `过程不满意` 或 `产物不满意`。

### 2.7 需要浏览器但未使用 `@浏览器`、`@chrome` 或 Safari

这是浏览器验收场景的强制 STOP。

进入本条判断前必须先完成真实能力探测：

- 如果本次需要可见浏览器验收，先读取并执行 `browser:control-in-app-browser` 的 bootstrap，使用 Codex in-app Browser 的 `iab` 后端；不要因为聊天里没有独立的 `@浏览器` 工具名就直接判不可用。
- 只有 `iab` bootstrap、连接或关键操作失败，才算 `@浏览器` 当前不可用；必须记录具体失败点。
- 默认降级策略下，`@浏览器` 不可用后再读取并执行 `chrome:control-chrome` 的 bootstrap，使用 Chrome extension 后端；不要因为聊天里没有独立的 `@chrome` 工具名就直接判不可用。
- 只有 Chrome extension bootstrap、连接或关键操作也失败，才允许继续判断 Safari 兜底是否可用。

命中条件：

- 当前任务需要浏览器验收。
- 实际既没有使用 `@浏览器`，也没有在 `@浏览器` 不可用时改用 `@chrome`，也没有在 `@chrome` 不可用时改用 Safari。
- 或者打算改用 Google Chrome、Chrome 或其他系统浏览器。

命中后必须立即停止，不要继续验收，不要给完成结论。

命中后的唯一合法处理：

- 明确说明当前需要使用 `@浏览器` 验收；如果 `@浏览器` 当前不可用、连接失败或能力不足，则改用 `@chrome` 继续验收；如果 `@chrome` 也不可用，再改用 Safari。
- 如果 `@浏览器`、`@chrome` 和 Safari 都不可用，直接告知用户当前被什么问题卡住。
- 不要改用 Google Chrome、Chrome 或其他系统浏览器继续验收，除非用户在这次对话里明确指定。

### 2.7.1 无头浏览器验收

当且仅当用户在本次验收里明确指定 `acceptance_browser_mode=headless` 时：

- 允许使用无头浏览器完成验收。
- 不再因为未使用 `@浏览器`、`@chrome` 或 Safari 而触发 `2.7`。
- 无头模式下仍然必须完成页面进入、关键操作、结果落点、刷新后保持、返回或切换后的再次进入、异常输入或空数据下的表现这类验证。
- 如果无头浏览器启动失败、页面打不开、关键交互无法执行或无法读取结果，同样按验收失败处理。
- 如果用户没有明确指定该模式，就继续按默认可见浏览器模式执行。

### 2.8 当前线程未挂载 `@浏览器` 能力

这是浏览器验收前的能力门禁。

这里的“未挂载”只能来自真实探测结果：已经按 `browser:control-in-app-browser` 说明尝试通过 Node REPL 连接 `iab` 后端，但当前线程没有 Browser 插件后端、连接失败，或关键验收操作无法执行。不要只凭当前工具列表里没有名为 `@浏览器` 的工具就命中本条。

命中条件：

- 当前任务需要浏览器验收。
- 当前这次对话实际没有 `@浏览器` 可调用能力，或已经明确出现“当前对话没有使用 `@浏览器` 的能力”这类提示。
- 当前不是无头浏览器验收模式。

命中后处理：

- 先说明当前线程未挂载 `@浏览器` 能力。
- 如果当前 `browser_policy=prefer_browser_allow_chrome_fallback`，先按 `chrome:control-chrome` 说明真实尝试连接 Chrome extension 后端并继续浏览器验收；如果 Chrome extension 也不可用，再用 Computer Use 操作 Safari 兜底。
- 如果当前 `browser_policy=browser_only`，立即停止，并明确要求切换到带 `@浏览器` 能力的新线程后再继续。
- 如果 `@chrome` 和 Safari 都不可用，再说明当前浏览器验收被什么问题卡住。
- 不要自行改用 Google Chrome、Chrome 或其他系统浏览器，除非用户在这次对话里明确指定。
- 如果是无头浏览器验收模式，按无头模式继续执行，不要把缺少 `@浏览器` 当成阻塞。

### 2.9 强制 `@浏览器` 模式下 Browser 不可用

这是浏览器验收场景的强制 STOP。

强制模式下也必须先真实尝试 `browser:control-in-app-browser` 的 `iab` bootstrap。只有 bootstrap、连接或关键操作失败后，才允许按本条停止；不要因为当前消息没有显式唤起 `@浏览器` 或工具列表里没有这个别名就提前停止。

命中条件：

- 当前任务需要浏览器验收。
- 当前 `browser_policy=browser_only`。
- 当前线程未挂载 `@浏览器` 能力，或 `@浏览器` 连接失败、能力不可用、无法完成本次验收。

命中后唯一合法处理：

- 明确说明这次验收被强制限定为只允许 `@浏览器`。
- 明确说明当前线程没有可用的 `@浏览器` 能力，或当前 `@浏览器` 被什么问题卡住。
- 明确要求用户新开一个带 Browser 插件的独立线程，并把首条消息写成浏览器验收任务。
- 不要降级到 `@chrome` 或 Safari。
- 不要改用 Google Chrome、Chrome 或其他系统浏览器。
- 不要继续给验收结论。

### 2.9.1 无头浏览器模式不可用

如果用户明确指定了 `acceptance_browser_mode=headless`，但无头浏览器无法启动、无法连接、页面无法打开，或关键交互无法执行：

- 立即停止无头验收。
- 明确说明当前卡在无头浏览器能力或启动问题上。
- 不要自动切回 `@浏览器`、`@chrome` 或 Safari，除非用户重新授权切换模式。
- 不要继续给验收结论。

### 2.10 代码已足以判 `未完成` 却仍试图启动服务或进入浏览器

这是验收分级前的强制 STOP。

命中条件：

- 当前已经进入验收阶段。
- 通过代码变更、测试结果、类型检查、命令行验证或直接相关文件内容，已经足以支持 `未完成` 结论。
- 后续仍打算启动本地服务、进入 `标准验收`、进入 `重验收`，或使用任何浏览器补页面证据。

命中后唯一合法处理：

- 立即停止后续服务启动和浏览器验收动作。
- 明确按代码侧证据直接输出 `未完成` 或继续组织 `未完成` 所需的验收结论字段。
- 不要再进入 `标准验收`。
- 不要再进入 `重验收`。
- 不要再使用 `@浏览器`、`@chrome` 或 Safari 补验证。

## 3. 输出级硬禁令

最终要发给用户、并且用户可能直接复制使用的输出级内容，必须先过这一层。输出级内容包括：

- 生成的提示词。
- `过程不满意`。
- `产物不满意`。
- 跟这三类内容紧挨着的简短结论补充。

这些内容里严禁出现：

- `稳定`
- `不稳定`
- `可靠`
- `不可靠`
- `收口`
- `收住`
- `落下来`
- `这一轮`
- `上一轮`
- `当前轮次`
- `同上`
- `（继续）`

如果一句话必须表达“基于这次结果继续修”，也要直接改写成当前看到的问题和要补的行为，不要借助跨次承接词。

输出级内容还有 3 条格式硬规则：

- `过程不满意` 不能以页面、按钮、文案、提示或内容显示英文作为理由；英文显示属于产物现象，只能放进 `产物不满意`。
- `过程不满意`、`产物不满意` 和 `提示词` 每一行在换行前都必须以标点结尾，优先用 `。`、`！`、`？`。
- 如果输出了 `下一轮提示词`，它必须以 `修复` 两个字开头。
- `过程不满意`、`产物不满意` 里禁止使用 `稳定`、`不稳定`、`可靠`、`不可靠` 这 4 个词；需要表达结果反复、行为飘忽、缺少确定性时，必须改写成具体可观察的现象、触发条件和业务影响。
- 只要 `过程不满意` 或 `产物不满意` 命中 `稳定`、`不稳定`、`可靠`、`不可靠` 任一词，当前输出立即视为无效，必须当场整段重写；不要带着命中的版本继续往下输出。

## 4. 总原则

- 只要进入验收，默认先做结果验收。
- 验收完成后，必须通过 `$solo-acceptance-results` 把结果回填到当前子项目目录的直接父目录里的独立结果 Excel；例如子项目是 `.../solo-6600021/6600021-codegen-2`，结果 Excel 就写到 `.../solo-6600021/solo-create-acceptance-results2.xlsx`；禁止把验收结果、Repo URL、Commit ID、Trae Session ID 或不满意原因写入 `solo-create-prompts.xlsx`。
- 结果 Excel 与生成提示词 Excel 必须分离；默认结果文件名是 `solo-create-acceptance-results2.xlsx`，第一次不存在时由 `$solo-acceptance-results` 新建。
- 验收默认目标是优先找出“未完成”的直接证据，不以替实现补理由，也不以凑出“已完成”结论为目标。
- 只要是前端页面、前端交互、可视化结果、浏览器流程或用户能点到的链路，默认按“需要浏览器严格实操”处理；除非代码侧已经足以直接判 `未完成`。
- 只要通过代码变更已经足以判定当前验收结论是 `未完成`，就必须当场停止在代码验收层，不要启动本地服务，不要进入任何浏览器验证，也不要再补做“确认一下页面”的动作。
- `@浏览器` 是默认优先使用的浏览器验收工具；执行时必须先按 `browser:control-in-app-browser` 连接 Codex in-app Browser 的 `iab` 后端。只有真实连接或操作失败后，才按 `@chrome` 降级；`@chrome` 必须按 `chrome:control-chrome` 连接 Chrome extension 后端。只有 Chrome extension 也真实不可用时，才允许再用 Computer Use 操作 Safari 兜底。
- 如果用户明确指定无头浏览器模式，则优先使用无头浏览器执行验收；无头模式不影响默认的结果验收顺序和文件回填要求。
- 如果用户在本次消息里明确要求“100% 使用 `@浏览器`”“只允许 `@浏览器`”“不要 `@chrome` 或 Safari 降级”，就把 `browser_policy` 设为 `browser_only`，并按 `2.9` 执行硬停。
- 不要把 Google Chrome、Chrome 或别的系统浏览器当成默认等价替代。
- 只要验收需要本地启动前端服务，就必须把“端口占用时顺延找可用端口启动”和“输出结论后停掉这次验收启动的服务”当成强约束执行，不能省略。
- 浏览器验收时必须按真实用户路径逐步点击、输入、切换、刷新、回退、重进，不要只看首屏，不要只凭截图，不要只做一次成功路径就结束。
- 只要出现任何关键行为缺失、状态不同步、交互歧义、文案与行为矛盾、刷新后失效、回跳异常、列表详情联动不一致、边界输入异常、空态异常、报错未兜底，默认判 `未完成`。
- 只有用户这次消息里真的贴了 Trae 过程信息，才叠加做过程验收。
- 如果结果没问题，但过程里明显有冗余思考、待办拆法很怪、语言跑偏、一直重复解释、执行节奏很乱，也不能直接算高质量完成。
- 如果用户没贴 Trae 过程信息，就不要凭空编造过程记录；但只要结果不满意，`过程不满意` 仍然必须按本文件的质量规则完整成立，不再要求“简短反推”。

### 4.1 提高 `@浏览器` 命中率

当后续明显会进入浏览器验收时，优先这样组织线程和首条消息：

- 尽量新开一个独立线程，不要把读代码、改代码、批量处理和浏览器验收全塞进同一条长上下文。
- 首条消息直接点名 `@浏览器`，不要只写“打开页面”或“帮我验收”。
- 如果目标是 100% 只用 `@浏览器`，首条消息里必须明确写“只允许 `@浏览器`，不要 `@chrome` 或 Safari 降级”。
- 首条消息里尽量同时给出可访问目标，例如 `http://localhost:3000`、`http://127.0.0.1:5173` 或明确的页面入口。
- 优先把任务表述成浏览器操作任务，例如“用 `@浏览器` 打开某地址并按真实用户路径验收某流程”。
- 默认策略下，如果当前线程已经提示缺少 `@浏览器` 能力，先确认是否已经真实尝试 `iab` bootstrap；如果没有，先尝试 `iab`。确认失败后再尝试 Chrome extension；只有 Chrome extension 和 Safari 都不可用时，再建议用户新开线程重试。
- 强制策略下，如果当前线程已经提示缺少 `@浏览器` 能力，也必须先确认是否已经真实尝试 `iab` bootstrap；确认失败后不要继续验收，直接要求用户新开一个带 Browser 插件的线程。
- 无头模式下，首条消息或验收上下文里要明确写出 `acceptance_browser_mode=headless`，避免和默认可见浏览器模式混淆。

## 7. 验收分级

验收是 route 内阶段，不是单独 route。进入验收前，先按 `2.1` 判断本次走 `带 Session 提交态` 还是 `无 Session 只验收态`。

- 如果通过代码变更已经足以支持 `未完成` 结论，即使任务本身是前端页面或强交互任务，也不要启动服务，不要进入浏览器，不要为了补截图、补复现或补正证据再做页面验证。

### 7.1 轻验收

- 项目是纯后端、脚本、CLI、服务端接口或数据处理任务。
- 当前环境起不来，但代码侧证据已经足够强。

必须做：

- 看相对基线改动。
- 读真正新增或继续修改的文件。
- 按需要运行后端测试、类型检查、接口相关命令或最小命令行验证。
- 只要本次轻验收已经能判 `未完成`，后续就不要再启动服务，不要再进入 `标准验收` 或 `重验收` 补浏览器证据。
- 直接给完成判断。

### 7.2 标准验收

标准验收只允许用于下面两类场景：

- 代码侧已经初步判断前端结果做对，但还缺少页面实操正证据，无法直接输出 `已完成`。
- 代码侧证据还不够，当前既不能直接判 `未完成`，也不能直接判 `已完成`，而本地可运行、确实需要结果确认。

只要不满足这两个前提之一，就不要进入标准验收。

进入这一路之前必须先排除一种情况：

- 如果代码侧已经足以判 `未完成`，即使用户提了页面、交互、浏览器或前端，也不能进入标准验收；必须留在轻验收直接给 `未完成`。
- 如果还试图进入标准验收，直接视为命中 `2.10`，立即停止后续服务启动和浏览器动作。

必须做：

- 看相对基线改动。
- 判断项目类型。
- 前端任务启动服务并打开页面；纯后端不走浏览器。
- 如果默认端口已被占用，必须继续往后顺延端口直到找到可用端口，再用该端口启动服务。
- 给出最终结论后，必须停掉这次验收阶段新启动的服务进程。
- 用 `@浏览器`、`@chrome` 或 Safari 至少走一遍核心路径，并补做最容易漏掉的后续动作验证。
- 给固定格式结论。

### 7.3 重验收

重验收只允许建立在一个前提上：

- 代码侧还不能直接判 `未完成`，并且任务价值主要落在真实页面交互结果，必须靠更重的实操覆盖来确认是否真的 `已完成`。

只有先满足这个前提，才允许再看下面的升级条件。

命中任一情况时升级到重验收：

- 代码侧初判为完成，但任务价值主要落在页面交互结果。
- 前端强交互任务。
- 路由、滚动、回跳、多步骤表单。
- 列表详情联动、筛选分页联动、跨区域联动。
- 用户明确要求用 Codex Browser 实操。
- 用户贴了 Trae 过程信息。
- 上一次已经证明“只看代码不够，必须真点一遍”。
- 用户明确要求“验收一定要特别严格”。

进入这一路之前也必须先排除一种情况：

- 如果代码侧已经足以判 `未完成`，不要因为“任务价值主要落在页面交互结果”或“用户要求特别严格”就升级到重验收；必须直接停在代码验收结论，不启动服务，不进入浏览器。
- 如果还试图升级到重验收，直接视为命中 `2.10`，立即停止后续服务启动和浏览器动作。

重验收必须额外执行：

- 默认先使用 `@浏览器`，并按 `browser:control-in-app-browser` 真实连接 `iab` 后端；如果 `iab` 当前不可用，则按 `chrome:control-chrome` 改用 Chrome extension 后端；如果 Chrome extension 也不可用，再用 Computer Use 操作 Safari；不要改用别的浏览器代替默认路径。
- 如果 `browser_policy=browser_only`，则必须只使用 `@浏览器`；当前线程没有 Browser 能力或 Browser 不可用时，按 `2.9` 停止。
- 如果默认端口已被占用，必须继续往后顺延端口直到找到可用端口，再用该端口启动服务。
- 先按主路径完整走一遍，再做至少 3 类补充检查：刷新后重试、切换数据或路由后重试、返回上一步再进入重试。
- 对表单、筛选、分页、弹窗、详情抽屉、标签切换、滚动加载、登录态依赖、草稿态依赖，命中哪个测哪个，不要因为首个路径成功就停止。
- 必须主动寻找反例，不要只记录成功路径。
- 只要有一个关键反例成立，就直接判 `未完成`。
- 给出最终结论后，必须停掉这次验收阶段新启动的服务进程。

### 7.4 浏览器规则

- 浏览器不是“代码初判未完成”时的默认动作；代码侧已经能判未完成，就直接给结论。
- 只要代码侧已经能判 `未完成`，不仅不要启动浏览器，也不要启动本地服务；浏览器规则只服务于“验证是否已完成”或“代码证据不足时补结果证据”，不能用来反向确认已经成立的 `未完成`。
- 纯后端项目、CLI 项目、数据处理项目、脚本项目不启动浏览器。
- 如果通过代码变更已经初步判断前端任务完成，必须启动服务并使用 `@浏览器`、`@chrome` 或 Safari 实操验证。
- 如果 `browser_policy=browser_only`，必须启动服务并使用 `@浏览器` 实操验证；不允许 `@chrome` 或 Safari 代替。
- 启动服务时如果预期端口已被占用，必须继续往后尝试新端口，直到找到可用端口；不要因为端口冲突直接放弃验收，也不要停掉别的项目正在使用的端口。
- 服务启动失败、页面起不来、关键链路无法进入时，前端任务不能判 `已完成`。
- 进入浏览器验收前，先按 `browser:control-in-app-browser` 真实尝试连接 `iab` 后端来确认当前线程是否有 `@浏览器` 能力；如果没有，默认策略按 `2.8` 先尝试 Chrome extension 后端，再按需要用 Computer Use 操作 Safari；强制策略按 `2.9` 停止。
- 需要浏览器时，默认优先使用 `@浏览器`。
- 需要浏览器时，不要自行改用 Google Chrome、Chrome 或其他系统浏览器。
- 如果 `@浏览器` 不可用、连接失败，或被能力边界客观卡住，默认策略先按 `chrome:control-chrome` 降级到 Chrome extension；如果 Chrome extension 也不可用或被能力边界客观卡住，再用 Computer Use 操作 Safari；强制策略按 `2.9` 停止。
- 如果 `@chrome` 和 Safari 都不可用或被客观卡住，再停止并说明阻塞原因。
- 当前这次验收只要新启动过本地服务，就必须在输出最终结论之后立即停掉该服务；这条是强约束，不要遗漏，也不要把服务留在后台。
- 浏览器验收时至少覆盖这 6 个检查点里与任务相关的部分：页面进入、关键操作、结果落点、刷新后保持、返回或切换后的再次进入、异常输入或空数据下的表现。
- 不能只打开页面看静态结果；只要任务涉及点击、输入、切换、弹层、滚动、异步加载、路由跳转、列表详情联动，就必须实操。
- 不能因为“看起来差不多”给 `已完成`；必须拿到明确正证据。
- 找不到明确正证据时，默认不输出 `已完成`；能确认缺陷就判 `未完成`，证据仍不足才判 `暂时无法判定完成`。

### 7.5 验收闭环提交规则

验收闭环分两种模式执行，不要混用：

- `带 Session 提交态`：用户已提供格式合法的 `Trae Session ID`，允许执行提交、push、带 `Commit ID` / `Trae Session ID` 的输出。
- `无 Session 只验收态`：用户没有提供合法 `Trae Session ID`，或明确表示这次不提供；只做验收，不做任何 git 提交和 push。

必须满足：

- 验收结论里的 `【提示词】` 必须严格对应“当前这次实际被验收的提示词”，不能回填别的轮次内容。
- 如果当前验收的是修复轮次结果，`【提示词】` 必须填写该次修复提示词，不能填写首轮提示词，也不能把首轮提示词和修复提示词拼在一起。
- 如果一次会话里先有首轮提示词，后又生成过修复提示词，后续所有围绕修复结果的验收、提交核对和最终输出，都以最新那条有效修复提示词作为当前验收依据。
- 如果当前是 `带 Session 提交态`，且工作区里存在围绕当前提示词的改动，不管最终结论是 `已完成`、`未完成` 还是 `暂时无法判定完成`，都必须先用合法 `Trae Session ID` 提交一次，再尝试 push 到远程，然后再输出验收结论。
- `带 Session 提交态` 下，用于 push 的提交 message 必须严格等于这个合法 `Trae Session ID`。
- `带 Session 提交态` 下，如果当前分支上还没有以该合法 `Trae Session ID` 为 message 的提交，先补提交再 push。
- `带 Session 提交态` 下，必须核对“当前有效 `Trae Session ID` = `git commit` message = 最终质检结论里的 `【Trae Session ID】`”；三者任一不一致，都不能继续给最终验收结论。
- `带 Session 提交态` 下，如果已经有同 message 的提交，但工作区里还有围绕当前提示词的新改动，默认先确认是否允许 amend；用户没有明确允许 amend 时，不要跳过“提交并 push”动作。
- 如果当前是 `无 Session 只验收态`，一定不能执行 `git commit`、`git push`、`git amend`，也不要因为工作区有改动就强行补提交流程。
- `Repo URL` 必须取当前仓库远端 `origin` 对应的 GitHub 仓库地址，统一规范化为 `https://github.com/owner/repo`，只输出最终 URL 本体，禁止带 `.git` 后缀。
- `带 Session 提交态` 下，`Commit ID` 必须填这个已 push 到远端的完整 40 位提交哈希。
- `带 Session 提交态` 下，`Trae Session ID` 必须原样回填当前上下文里可用、且通过格式校验的原始值。
- `无 Session 只验收态` 下，最终输出里不要展示 `Commit ID` 和 `Trae Session ID`。
- 不管当前是哪种验收模式，最终输出前都必须统计“这次修改的文件个数”，并把它作为单独字段输出。
- `修改文件个数` 必须按当前这次验收实际涉及的修改文件集合统计，输出整数值。
- 统计口径要前后一致：优先按当前工作区里围绕本次提示词的已修改文件集合统计；如果当前是 `带 Session 提交态` 且改动已经提交，也可以按这次被验收提交对应的文件集合统计，但最终只允许输出一个整数。
- 统计时不要把 `node_modules`、构建产物、日志、截图、临时文件或与本次提示词无关的环境文件混进来；只统计围绕这次提示词实际改动的源码、配置、样式、测试、脚本和文档文件。
- 如果当前是 `带 Session 提交态`，且因为远端不可达、权限不足、仓库未配置 upstream 或其他 push / remote 问题导致没法正常 push，立即命中 `2.6`；先告知用户问题并恢复正常推送，恢复前不要输出验收结果。
- 只要没有命中 `2.6`，在最终验收模板发给用户前必须调用 `$solo-acceptance-results` 回填结果 Excel；回填不改变原有聊天输出字段，也不要在最终模板里新增 Excel 路径或回填状态。
- `$solo-acceptance-results` 必须写当前子项目目录的直接父目录里的独立结果 Excel，默认文件名 `solo-create-acceptance-results2.xlsx`；例如 `.../solo-6600021/6600021-codegen-2` 必须写到 `.../solo-6600021/solo-create-acceptance-results2.xlsx`；如果第一次不存在就新建，表头必须和 `/Users/fengyin/项目/字节/solo/数据中转/solo-create-acceptance-results2.xlsx` 一致。
- 回填给 `$solo-acceptance-results` 的 `Repo ID` 必须是 `ybl-<数字编号>-<序号>` 标准格式；不能从项目目录名解析出标准值时，必须显式传入标准 `repo_id`，否则回填脚本会失败，禁止原样写入目录名。
- 回填输入必须来自当前 `acceptance_context`、`final_output` 和 `$solo-dissatisfaction` 结果，不要重新改写验收结论。
- 如果 `completion != 已完成`，回填输入还必须携带 `$solo-dissatisfaction` 校验所需的 `next_prompt`、过程证据、产物证据、模型自身责任依据和 `environment_issue_excluded=true`；回填脚本会再次运行校验，校验失败时禁止写入 Excel。
- 如果 `$solo-acceptance-results` 回填失败，先修复回填问题并重试；禁止改写到 `solo-create-prompts.xlsx`，禁止静默跳过回填后结束验收流程。

### 7.6 过程验收触发条件

只有当用户这次消息里真的贴了 Trae 过程信息，才做过程验收。

过程验收重点方向：

- 理解 prompt 是否偏了。
- 目标是否明确。
- 推理过程是否精简。
- 输出是否正常。
- 任务规划是否合理。
- 工具使用是否合理高效。
- 操作过程是否合理。
- 有没有高危操作或越界。
- 总结是否正确。
- 有没有虚假完成。

如果用户没贴过程内容，就不要凭空补过程记录。但只要结果不满意，`过程不满意` 仍然必须写成合规结论。写法只能二选一：

- 模型行为评价。
- 更深一层的代码级 / 业务逻辑级根因。

不要把业务 Bug 直接换壳写成“实现上少了校验”“没有核对结果”“没有处理好”“没有保住状态”这类空话。

## 8. 完成判断

完成状态只允许这 3 种：

- `已完成`
- `未完成`
- `暂时无法判定完成`

判断标准优先看：

- 核心行为是不是真的存在。
- 有没有明显交互问题。
- 有没有破坏原来的阅读流或使用流。
- 如果任务和缩放、筛选、切换条目有关，顺手验证这些后续动作。

两条硬规则：

- 如果当前是 `带 Session 提交态`，只有结果满足验收要求，且对应代码已经以合法 `Trae Session ID` 为 message 提交并 push 到远程，才允许输出 `已完成`。
- 如果当前是 `无 Session 只验收态`，`已完成` 只取决于验收证据本身；不要因为没有 `Trae Session ID`、没有提交、没有 push 就阻塞完成结论。
- 只要前端任务没有经过 `@浏览器`、`@chrome` 或 Safari 严格实操，或实操覆盖不到关键链路，就不能输出 `已完成`。
- 只要验收证据同时存在“像是做了”和“仍可能没做完”的冲突，默认按 `未完成` 处理，不要替实现补乐观解释。

### 11.3 输出前自检

最终输出前，必须逐项自检：

- route 是否唯一且与执行过程一致。
- 当前输出是否命中 `Hard Stops`。
- 当前是否已经正确判定本次属于 `带 Session 提交态` 还是 `无 Session 只验收态`。
- 当前 `browser_policy` 是否已经按本次用户要求正确设定。
- 当前是否已经正确识别本次是默认可见浏览器模式，还是 `acceptance_browser_mode=headless`。
- 如果本次已经通过代码侧证据判定 `未完成`，是否确实已经命中并遵守 `2.10`，没有启动本地服务，也没有进入任何浏览器验收。
- 当前这次浏览器验收是否实际使用了 `@浏览器`、`@chrome`、Safari，或用户指定的无头浏览器模式；如果是强制策略，是否已经严格只使用 `@浏览器`；如果不满足，是否已经按 `2.7`、`2.7.1`、`2.9` 或 `2.9.1` 停止。
- 如果当前验收阶段新启动了本地服务，是否已经在端口被占用时自动顺延到可用端口，而不是卡在冲突端口。
- 如果当前验收阶段新启动了本地服务，是否已经在准备输出最终结论前安排并执行停服，而不是把服务遗留在后台。
- 当前这次验收是否已经通过 `$solo-acceptance-results` 回填独立结果 Excel，且没有把验收结果写入 `solo-create-prompts.xlsx`。
- `$solo-acceptance-results` 是否返回 `ok: true`，且写入文件不是生成提示词 Excel。
- 输出级内容是否含有禁词。

- 如果 `solo-dissatisfaction` 或校验脚本判定当前版本无效，是否已经整段废弃并重写，而不是只替换几个词继续沿用原句结构。
- `过程不满意`、`产物不满意`、`提示词` 每行是否以标点结尾。
- `修改文件个数` 是否已经统计完成，且是否紧跟在 `【提示词】` 后面输出。
- 如果输出了 `下一轮提示词`，它是否以 `修复` 开头。

### 12.3 多项目并行

- 一个项目只对应一个工作目录。
- 一个项目只对应一组端口。
- 一个项目只对应自己的基线记录。
- 一个项目只对应内置浏览器里的一组固定标签页。

如果两个项目端口相同，默认从原端口 `+1` 往后找可用端口，不要停已有服务。

只要本次验收为了当前项目新启动过服务，输出结论之后必须把这次新启动的服务停掉；不要因为端口是后来顺延找到的，就把该进程留在后台。

如果两个项目依赖同一个登录态、同源浏览器存储、同一套会互相污染的数据环境，或当前没法可靠区分标签页，就不要直接并行验收。
