# 项目出题循环

这套流程的目标只有一个：更快地产出贴项目、能直接复制、能持续迭代的提示词和验收结论。

本 skill 必须按“先定路由，再过门禁，再执行单一路径，再按唯一输出模板返回”的顺序运行。不要边做边换路，不要把说明性文字当成可选建议。

## 0. 执行契约

### 0.1 强制状态机

每次运行都必须先选且只选一个 route。合法 route 只有这 6 个：

- `已投递验收路`
- `首轮半严路`
- `后续快路`
- `批量建仓路`
- `批量首轮路`
- `批量投递路`

没有选出 route 之前：

- 不要扫仓库。
- 不要打开 Trae。
- 不要读手册。
- 不要启动服务。
- 不要生成提示词。
- 不要给验收结论。

### 0.2 执行前自检

进入动作前，先在内部完成这 5 项自检：

- `route` 是什么。
- 当前是否是验收场景。
- 当前是单项目还是父目录批量场景。
- 当前是否已经收到本次对话里的合法 `Trae Session ID`，以及这次验收要走“带 Session 提交态”还是“无 Session 只验收态”。
- 当前最终允许输出哪些字段。

如果其中任一项无法确定，优先按本文件的 `Hard Stops` 执行；仍然无法确定时，只允许向用户提出一个最小澄清问题。

### 0.3 单项目起始动作

只要 route 是单项目流程，就必须先运行：

- `python3 /Users/fengyin/.codex/skills/solo-create/scripts/ensure_trae_project_open.py --project-path .`

这一步优先级高于扫仓库、记基线、查历史、读手册、启动项目和浏览器验收。不要后移，不要懒触发，不要省略。

批量首轮路不执行这一步。批量投递路只在真正投递某个子项目时执行这一步。

### 0.4 核心数据格式

除了流程顺序，核心数据也必须先匹配 [data-contract.md](/Users/fengyin/.codex/skills/solo-create/references/data-contract.md) 里的 schema，再进入 route 判断或最终输出。

必须使用并遵守这 6 组数据对象：

- `run_context`
- `project_lookup`
- `workbook_row`
- `acceptance_context`
- `dissatisfaction_block`
- `final_output`

执行要求：

- 进入动作前，先在内部把当前上下文收束成 `run_context`。
- 读取 `locate-project` 结果时，先映射成 `project_lookup`。
- 读取或更新 Excel 时，逐行按 `workbook_row` 理解，不要自造列。
- 进入验收时，先把上下文收束成 `acceptance_context`，再判断 `已完成 / 未完成 / 暂时无法判定完成`。
- 生成不满意原因时，先满足 `dissatisfaction_block` 的字段和角色分工。
- 最终返回给用户时，只允许输出 `final_output` 允许的形态。
- 只要涉及 `Trae Session ID`，先按 [data-contract.md](/Users/fengyin/.codex/skills/solo-create/references/data-contract.md) 里的 `0. Trae Session ID 格式` 校验；不合法就按“未收到合法 Session ID”处理。
- 只有本次验收明确走“带 Session 提交态”时，才检查 3 个环节完全一致：当前有效值、`git commit` message、最终质检结论输出值。

如果本文件与 `data-contract.md` 的字段定义冲突，一律以 `data-contract.md` 为准。

## 1. Route Selection

按下面顺序做 route 判定。命中即停，不再继续往下匹配。

### 1.1 已投递验收路

当满足下面全部条件时，必须进入 `已投递验收路`：

- 当前目录是单个工程。
- 运行：
  - `python3 /Users/fengyin/.codex/skills/solo-create/scripts/batch_prompt_workbook.py locate-project --project .`
- 返回 `found: true`。
- 记录里的 `提示词` 非空。

命中后：

- 不再生成首轮提示词。
- 不再重复历史验重。
- 不再重复读任务手册来出题。
- 把 Excel 里的 `提示词` 当成已发送给 Trae 的任务依据。

### 1.2 首轮半严路

当满足任一条件时，进入 `首轮半严路`：

- 用户明确要求“单个标准完整流程”。
- 用户明确要求“重新生成首轮提示词”。
- 用户明确要求“从头生成提示词”。
- 当前是单工程，且 `locate-project` 返回 `found: false`，且用户没有说明它属于批量 Excel 子项目。

### 1.3 批量建仓路

当用户要求在父目录下把一个源代码仓库批量复制成多个 GitHub 仓库时，进入 `批量建仓路`。典型表达包括：

- 建多个仓库。
- 复制仓库。
- GitHub 批量建仓。
- 初始化多个源码一样的仓库。

### 1.4 批量首轮路

当用户要求在父目录下遍历多个子文件夹、批量生成提示词或写入 Excel 时，进入 `批量首轮路`。

如果父目录下还只有一个源仓库，且用户的真实目标是先得到多个源码相同的 GitHub 仓库，不要进入批量首轮路，必须改走 `批量建仓路`。

### 1.5 批量投递路

当父目录下已存在 `solo-create-prompts.xlsx`，且用户给出编号范围、前几个、打开对应 Trae 窗口或要求自动发送时，进入 `批量投递路`。

### 1.6 后续快路

当已经出过一次题，用户是在补问题、修问题、继续改结果时，进入 `后续快路`。

### 1.7 验收分流

当用户回来让继续、看结果、验收、生成完了时：

- 如果已经命中 `已投递验收路`，留在该 route 内执行验收。
- 如果当前处理的是首轮提示词后的结果检查，仍然留在对应单项目 route 内执行验收。
- 不要新造第 7 条 route。验收只是 route 内的阶段，不是独立 route。

## 2. Hard Stops

下面这些门禁优先级最高。命中后必须立刻停止主流程，只允许输出指定内容。

### 2.1 验收缺少合法 Trae Session ID

这不再是默认 STOP，而是验收模式分流。

只要进入验收阶段，就先判断本次是：

- `带 Session 提交态`：用户已提供格式合法的 `Trae Session ID`，允许按本文件的提交 / push / 回填完整链路执行。
- `无 Session 只验收态`：用户没有提供合法 `Trae Session ID`，或明确表示这次不提供。

进入 `无 Session 只验收态` 时必须满足：

- 继续正常做代码验收、浏览器验收和最终结论输出。
- 一定不能执行 `git commit`、`git push`、`git amend`，也不要为了凑齐字段去生成或代填 `Trae Session ID`。
- 最终验收结果里不要展示 `Trae Session ID` 和 `Commit ID`。
- 如果当前项目属于 Excel 管理子项目，`验收Commit ID`、`验收Trae Session ID` 允许留空，不因此阻塞验收结果输出。

只有当用户已经提供格式合法的 `Trae Session ID` 时，才允许进入 `带 Session 提交态`。

### 2.2 已投递验收路但 Excel 状态不允许继续

如果 `locate-project` 命中后，记录状态是：

- `已发送`：可以直接进入验收；有合法 `Trae Session ID` 就走 `带 Session 提交态`，没有就走 `无 Session 只验收态`。
- `已生成` 但不是 `已发送`：必须先提醒“这条提示词可能还没有投递”；但这不再阻塞验收，仍可按当前代码和结果进入验收。

这时不要直接给验收结论。

### 2.3 批量投递时 Trae 未聚焦到目标项目

如果当前窗口标题不包含目标子文件夹名称，或无法确认聊天输入框已经聚焦：

- 记录该条失败并跳过。
- 不要更新 Excel 状态。
- 不要把提示词发到不确定输入区域。

### 2.4 无法判定 source 项目编号

批量建仓路里，如果源仓库目录名提取不出连续数字，且用户没有指定 `--source-number`：

- 立即停止。
- 只要求用户补 `--source-number`。
- 不要自己编编号。

### 2.5 路由无法确定

如果按 `Route Selection` 顺序执行后仍然无法确定 route：

- 只允许向用户提出一个最小澄清问题。
- 不要提前扫仓库或生成内容。

### 2.6 验收阶段 push / remote 阻塞

如果验收阶段出现下面任一情况：

- 远端不可达。
- 远端权限不足。
- 仓库未配置可用 upstream。
- `git push` 失败且当前无法恢复到正常推送。

本条只适用于 `带 Session 提交态`。如果当前是 `无 Session 只验收态`，根本不要进入 push / remote 流程，也不要因为 remote 状态阻塞验收输出。

必须立即停止验收结论输出。

这时只允许：

- 明确告知用户当前的 push / remote 问题是什么。
- 说明必须先恢复正常推送，之后才能继续给验收结果。

这时禁止：

- 输出 `任务完成情况`。
- 输出 `不满意原因`。
- 输出 `下一轮提示词`。
- 把 push / remote 阻塞写进 `过程不满意` 或 `产物不满意`。

### 2.7 需要浏览器但未使用 `@浏览器`

这是浏览器验收场景的强制 STOP。

命中条件：

- 当前任务需要浏览器验收。
- 实际没有使用 `@浏览器`。
- 或者打算改用 Google Chrome、Chrome、Safari 或其他系统浏览器。

命中后必须立即停止，不要继续验收，不要给完成结论。

命中后的唯一合法处理：

- 明确说明当前需要使用 `@浏览器` 验收。
- 如果 `@浏览器` 当前不可用、连接失败或能力不足，直接告知用户当前被什么问题卡住。
- 除非用户在这次对话里明确授权降级到指定浏览器，否则不要改用 Google Chrome、Chrome、Safari 或其他系统浏览器继续验收。

### 2.8 当前线程未挂载 `@浏览器` 能力

这是浏览器验收前的能力门禁。

命中条件：

- 当前任务需要浏览器验收。
- 当前这次对话实际没有 `@浏览器` 可调用能力，或已经明确出现“当前对话没有使用 `@浏览器` 的能力”这类提示。

命中后处理：

- 直接说明当前线程未挂载 `@浏览器` 能力，当前不能继续浏览器验收。
- 明确建议用户新开一个以浏览器验收为主的线程，并在首条消息里直接点名 `@浏览器` 和目标地址或页面。
- 没有用户明确授权前，不要自行改用 Google Chrome、Chrome、Safari 或其他系统浏览器。
- 不要继续给 `已完成`、`未完成` 或 `暂时无法判定完成` 结论。

### 2.9 验收结论需要回填但 Excel 回填失败

这是批量 Excel 子项目验收场景的强制 STOP。

命中条件：

- 当前项目属于 `solo-create-prompts.xlsx` 管理的子项目。
- 当前已经形成 `已完成`、`未完成` 或 `暂时无法判定完成` 的最终验收结论。
- 但还没有把验收信息成功回填到 Excel，或回填后复查发现字段值和最终结论不一致。
- 或者回填后只通过了单一路径校验，仍然不能确认目标主提示词 / 修复提示词那一行已经正确落盘。

命中后必须立即停止，不要继续输出验收模板。

命中后的唯一合法处理：

- 明确说明当前卡在 Excel 验收回填。
- 说明必须先把验收信息成功写回对应行，之后才能继续输出最终验收结果。
- 明确说明“回填是强指令，未完成回填视为验收未完成”。
- 不要把回填失败写进 `过程不满意` 或 `产物不满意`。

## 3. 输出级硬禁令

最终要发给用户、并且用户可能直接复制使用的输出级内容，必须先过这一层。输出级内容包括：

- 生成的提示词。
- `过程不满意`。
- `产物不满意`。
- 跟这三类内容紧挨着的简短结论补充。

这些内容里严禁出现：

- `稳定`
- `不稳定`
- `可靠`
- `不可靠`
- `收口`
- `收住`
- `落下来`
- `这一轮`
- `上一轮`
- `当前轮次`
- `同上`
- `（继续）`

如果一句话必须表达“基于这次结果继续修”，也要直接改写成当前看到的问题和要补的行为，不要借助跨次承接词。

输出级内容还有 3 条格式硬规则：

- `过程不满意` 不能以页面、按钮、文案、提示或内容显示英文作为理由；英文显示属于产物现象，只能放进 `产物不满意`。
- `过程不满意`、`产物不满意` 和 `提示词` 每一行在换行前都必须以标点结尾，优先用 `。`、`！`、`？`。
- 如果输出了 `下一轮提示词`，它必须以 `修复` 两个字开头。
- `过程不满意`、`产物不满意` 里禁止使用 `稳定`、`不稳定`、`可靠`、`不可靠` 这 4 个词；需要表达结果反复、行为飘忽、缺少确定性时，必须改写成具体可观察的现象、触发条件和业务影响。
- 只要 `过程不满意` 或 `产物不满意` 命中 `稳定`、`不稳定`、`可靠`、`不可靠` 任一词，当前输出立即视为无效，必须当场整段重写；不要带着命中的版本继续往下输出。

## 4. 总原则

- 只要进入验收，默认先做结果验收。
- 验收默认目标是优先找出“未完成”的直接证据，不以替实现补理由，也不以凑出“已完成”结论为目标。
- 只要是前端页面、前端交互、可视化结果、浏览器流程或用户能点到的链路，默认按“需要 `@浏览器` 严格实操”处理；除非代码侧已经足以直接判 `未完成`。
- `@浏览器` 是唯一默认允许的浏览器验收工具；不要把 Google Chrome、Chrome、Safari 或别的系统浏览器当成等价替代。
- `@浏览器` 是否可用取决于当前线程运行时是否实际挂载了 `Browser` 插件能力；规则要求必须使用，不等于当前线程一定已经具备该能力。
- 浏览器验收时必须按真实用户路径逐步点击、输入、切换、刷新、回退、重进，不要只看首屏，不要只凭截图，不要只做一次成功路径就结束。
- 只要出现任何关键行为缺失、状态不同步、交互歧义、文案与行为矛盾、刷新后失效、回跳异常、列表详情联动不一致、边界输入异常、空态异常、报错未兜底，默认判 `未完成`。
- 只有用户这次消息里真的贴了 Trae 过程信息，才叠加做过程验收。
- 如果结果没问题，但过程里明显有冗余思考、待办拆法很怪、语言跑偏、一直重复解释、执行节奏很乱，也不能直接算高质量完成。
- 如果用户没贴 Trae 过程信息，就不要凭空编造过程记录；但只要结果不满意，`过程不满意` 仍然必须按本文件的质量规则完整成立，不再要求“简短反推”。

### 4.1 提高 `@浏览器` 命中率

当后续明显会进入浏览器验收时，优先这样组织线程和首条消息：

- 尽量新开一个独立线程，不要把读代码、改代码、批量处理和浏览器验收全塞进同一条长上下文。
- 首条消息直接点名 `@浏览器`，不要只写“打开页面”或“帮我验收”。
- 首条消息里尽量同时给出可访问目标，例如 `http://localhost:3000`、`http://127.0.0.1:5173` 或明确的页面入口。
- 优先把任务表述成浏览器操作任务，例如“用 `@浏览器` 打开某地址并按真实用户路径验收某流程”。
- 如果当前线程已经提示缺少 `@浏览器` 能力，默认建议用户新开线程重试，不要在同一线程里反复硬试。

## 5. 单项目路径

### 5.1 标准完整流程

仅当用户明确要求从头生成首轮提示词时，按下面固定顺序执行：

1. 先执行 `ensure_trae_project_open.py`。
2. 进入 `首轮半严路` 生成首轮提示词。
3. 最终只输出提示词正文，不要夹带别的说明。
4. 同一次执行内直接进入验收阶段。
5. 如果验收未通过，切到 `后续快路` 生成修复提示词。

### 5.2 已投递验收路

进入单工程目录后，先运行：

- `python3 /Users/fengyin/.codex/skills/solo-create/scripts/batch_prompt_workbook.py locate-project --project .`

如果返回 `found: true`，且记录里 `提示词` 非空：

- 不再生成首轮提示词。
- 不再重复历史验重。
- 不再重复读任务手册来出题。
- 把 Excel 里的 `提示词` 当成该项目已发送给 Trae 的任务依据。
- `状态` 为 `已发送` 时，有合法 `Trae Session ID` 就走 `带 Session 提交态`，没有就走 `无 Session 只验收态`。
- `状态` 为 `已生成` 但不是 `已发送` 时，先提醒可能还没有投递；但不再要求必须等待用户发送合法 `Trae Session ID` 后才能进入验收。
- 验收没过时，按 `后续快路` 生成修复提示词。

如果返回 `found: false`：

- 说明当前工程不是批量 Excel 管理的子项目。
- 除非用户另有说明，否则退回 `标准完整流程`。

### 5.3 首轮半严路

首轮不走最重流程，但也不能只凭经验出题。必须按下面顺序执行：

1. 先执行 `ensure_trae_project_open.py`。
2. 轻扫仓库，优先用：
   - `rg --files`
   - `git status --short`
   - `git diff --stat`
3. 只读和当前切入点直接相关的文件。
4. 判断任务类型；明确就直走，不明确才让用户选。
5. 运行历史验重：
   - `python3 /Users/fengyin/.codex/skills/solo-create/scripts/query_prompt_history.py --project-path . --task-type "<任务类型>"`
6. 按任务类型只读命中的一本手册，不要把全部手册都读一遍。
7. 尝试记基线：
   - 优先运行仓库里的 `scripts/record_round_baseline.py`
   - 没有再回退到 skill 自带脚本
   - 如果缺失或失败，要明确告知当前是降级处理，后续验收只能基于累计改动判断
8. 生成符合“首轮提示词要求”的中文提示词。
9. 执行输出级禁词检查。
10. 记录提示词历史：
    - `python3 /Users/fengyin/.codex/skills/solo-create/scripts/record_prompt_history.py --project-path . --task-type "<任务类型>" --angle "<这次的切入点>" --prompt "<最终提示词>"`

首轮去重规则必须执行：

- 重点看同前缀、同类型、同 angle 的历史。
- 如果 `matched_count` 大于 `0`，优先换 angle，不要只改几个词。
- 如果当前类型是 `缺陷修复`，首轮优先找两个彼此有关联的问题，不要总出单点小修。
- 不要沿用“新增 xxx 功能，支持……并……”这类高频骨架反复换皮。
- 不要把历史提示词的动作链顺序原样保留后只替换对象词；发现主句结构接近时，直接换整句组织方式。

首轮提示词必须满足：

- 必须贴当前仓库。
- 必须是用户可感知的问题或需求。
- 默认用中文。
- 不要出现文件路径、函数名、类名、实现指挥。
- 不要写成任何项目都能套用的通用模板。
- 如果任务类型是 `代码生成`，默认从新增能力本身起笔，优先写成 `新增 xxx 功能，……`。
- 不要用“现在只有……，希望新增……”这种把现状铺在前面的句式。
- 只有不补一句背景就无法理解新增能力时，才允许补极短业务上下文。
- 如果一句提示词删掉“目前 / 现在 / 已有 / 现有 / 原来”后仍然成立，默认删掉再输出。

类型判断默认规则：

- 新增一块用户能独立感知的新能力、新入口或新流程，默认按 `代码生成`。
- 在已有能力上补规则、补交互、补展示，或在原流程上往前推一点，默认按 `功能迭代`。
- 不要因为题面里提到了某个现有页面、现有模块或现有入口，就把本该是 `代码生成` 的题写成 `功能迭代`。

### 5.4 后续快路

后续修复提示词默认走更短链路，不再重复首轮那套重动作。必须执行：

1. 先执行 `ensure_trae_project_open.py`。
2. 先看相对基线改动。
3. 先看上次验收确认过的问题点。
4. 只围绕已确认问题生成修复提示词。
5. 发给用户前，再做一次输出级禁词检查。

默认跳过：

- 不再重复做历史验重。
- 不再重复做任务类型选择，除非任务性质明显变了。
- 不再重读整本手册，除非输出风格明显跑偏。
- 不再重扫整仓库，只补读直接相关文件。

命中任一情况时，回到更严路径：

- 修复范围明显扩散，不再只是补问题。
- 用户中途换了需求方向。
- 基线缺失，累计改动已经看不清边界。
- 连续两次验收都发现问题描述和实际改动对不上。

修复提示词必须满足：

- 只写已经验出来的问题。
- 不要把最初大需求整段重写一遍。
- 必须以 `修复` 两个字开头。
- `修复` 后面必须立刻接当前要修的问题对象或用户可感知问题。
- 直接描述要补的行为结果，不要写过程指挥。
- 不要和这次 `过程不满意` 形成近似复述。
- 不要和这次 `产物不满意` 形成近似复述。
- 不要和本项目历史修复提示词只做同义词替换。

### 6.1 批量建仓路

当父目录初始化时只有一个源代码仓库，而用户要求批量复制成多个 GitHub 仓库时，走这条路。

执行前必须确认：

- 真实建仓前必须先 dry-run。

默认创建 43 个仓库：

- `功能迭代` 20 个。
- `代码理解` 1 个。
- `代码重构` 1 个。
- `工程化` 1 个。

默认命名格式：

- `<源项目编号>-<任务类型标识>-<序号>`
- `序号` 必须在所有任务类型之间全局递增。

ASCII 任务标识固定为：

- `代码生成` -> `codegen`
- `功能迭代` -> `feature`
- `代码理解` -> `understand`
- `代码重构` -> `refactor`
- `工程化` -> `engineering`

源项目编号优先从源仓库目录名中提取连续数字；提取不到时按 `2.4` 停止。

建仓命令：

- dry-run：
  - `python3 /Users/fengyin/.codex/skills/solo-create/scripts/create_batch_github_repos.py --parent "<父目录>" --source "<源仓库目录>" --dry-run`
- 真实创建：
  - `python3 /Users/fengyin/.codex/skills/solo-create/scripts/create_batch_github_repos.py --parent "<父目录>" --source "<源仓库目录>"`

常用参数：

- `--owner "<GitHub 用户或组织>"`
- `--visibility public|private|internal`
- `--source-number "<编号>"`
- `--codegen-count 20`
- `--feature-count 20`
- `--understand-count 1`
- `--refactor-count 1`
- `--engineering-count 1`
- `--gh-bin "<官方 gh 路径>"`
- `--git-user-name "<提交用户名>"`
- `--git-user-email "<提交邮箱>"`
- `--allow-dirty`

脚本行为约束：

- 为每个目标仓库创建同名本地子目录。
- 复制源仓库工作区内容，但不复制源仓库的 `.git`。
- 在每个目标目录重新 `git init`、提交初始代码。

- 使用 `gh auth token` 通过临时 HTTPS 凭据 push 初始提交。
- push 后把 `origin` 和 upstream 清理成不含 token 的普通 HTTPS 地址。
- 如果本地目标目录已存在且远端也存在，尝试补推缺失的远端分支。
- 如果本地目录或远端仓库已存在，默认不删除、不覆盖源码内容。
- 输出 JSON 汇总，包含 `planned`、`created`、`skipped`、`failed`。

建仓完成后必须核对：

- `failed_count` 必须是 `0`。
- 抽查本地目录的 `git remote get-url origin`，必须是不含 token 的普通 `https://github.com/<owner>/<repo>.git`。
- 用 `gh api repos/<owner>/<repo>/branches/<branch>` 或等价命令核对远端分支；默认 43 个仓库都必须有 `main`。

失败时处理：

- 先停掉残留的 `create_batch_github_repos.py` 或 `gh repo create` 进程。
- 如果只是 push 失败且远端已创建，不要删除远端；直接重跑同一条脚本补推。
- 如果本地目录用了错误身份或错误命名，确认源仓库不在删除范围内后，删除本次新建的目标目录再重新 dry-run。
- 如果 `.git/config` 里残留 `x-access-token` 或 `gho_`，必须先清理掉再继续。

建仓成功后，默认衔接：

## 7. 验收分级

验收是 route 内阶段，不是单独 route。进入验收前，先按 `2.1` 判断本次走 `带 Session 提交态` 还是 `无 Session 只验收态`。

最高优先级判断：

- 如果通过代码变更已经能明确判断结果没做对，直接判 `未完成`，不要启动浏览器确认。
- 如果通过代码变更已经初步判断结果做对了，不能直接判 `已完成`；必须启动服务并使用 `@浏览器` 严格实操后，才能给 `已完成`。
- 如果是纯后端项目，不启动浏览器；优先通过代码、接口逻辑、数据流、测试或命令行结果判断。
- 如果是前端页面或交互任务，只要代码侧初判为完成，就默认进入浏览器验证。
- 只有代码证据不够、且任务确实依赖前端交互或页面表现时，才允许在“未完成”和“暂时无法判定完成”之间继续补浏览器证据。
- 只要浏览器实操里有一步无法稳定复现预期结果，或需要主观替实现解释，默认直接判 `未完成`。

### 7.1 轻验收

命中任一情况即可：

- 代码改动已经足够明确说明结果没做对。
- 代码变更已经覆盖任务关键路径，继续启动浏览器不会增加判断依据。
- 项目是纯后端、脚本、CLI、服务端接口或数据处理任务。
- 当前环境起不来，但代码侧证据已经足够强。

必须做：

- 看相对基线改动。
- 读真正新增或继续修改的文件。
- 按需要运行后端测试、类型检查、接口相关命令或最小命令行验证。
- 直接给完成判断。

### 7.2 标准验收

只有代码侧已经初步判断前端结果做对，或代码侧证据不够、但本地可运行且需要结果确认时，才走标准验收。

必须做：

- 看相对基线改动。
- 判断项目类型。
- 前端任务启动服务并打开页面；纯后端不走浏览器。
- 用 `@浏览器` 至少走一遍核心路径，并补做最容易漏掉的后续动作验证。
- 给固定格式结论。

### 7.3 重验收

命中任一情况时升级到重验收：

- 代码侧初判为完成，但任务价值主要落在页面交互结果。
- 前端强交互任务。
- 路由、滚动、回跳、多步骤表单。
- 列表详情联动、筛选分页联动、跨区域联动。
- 用户明确要求用 Codex Browser 实操。
- 用户贴了 Trae 过程信息。
- 上一次已经证明“只看代码不够，必须真点一遍”。
- 用户明确要求“验收一定要特别严格”。

重验收必须额外执行：

- 必须使用 `@浏览器`，不要用别的浏览器代替默认路径。
- 先按主路径完整走一遍，再做至少 3 类补充检查：刷新后重试、切换数据或路由后重试、返回上一步再进入重试。
- 对表单、筛选、分页、弹窗、详情抽屉、标签切换、滚动加载、登录态依赖、草稿态依赖，命中哪个测哪个，不要因为首个路径成功就停止。
- 必须主动寻找反例，不要只记录成功路径。
- 只要有一个关键反例成立，就直接判 `未完成`。

### 7.4 浏览器规则

- 浏览器不是“代码初判未完成”时的默认动作；代码侧已经能判未完成，就直接给结论。
- 纯后端项目、CLI 项目、数据处理项目、脚本项目不启动浏览器。
- 如果通过代码变更已经初步判断前端任务完成，必须启动服务并使用 `@浏览器` 实操验证。
- 服务启动失败、页面起不来、关键链路无法进入时，前端任务不能判 `已完成`。
- 进入浏览器验收前，先确认当前线程是否真的挂载了 `@浏览器` 能力；如果没有，按 `2.8` 停止，不要继续尝试给结论。
- 需要浏览器时，默认优先使用 `@浏览器`。
- 需要浏览器时，不要自行改用 Google Chrome、Chrome、Safari 或其他系统浏览器。
- 如果 `@浏览器` 不可用、连接失败，或被能力边界客观卡住，默认直接停止并说明阻塞原因，不要自动降级。
- 只有用户在当前对话里明确授权“可以改用某个指定浏览器”时，才允许一次性降级；没有明确授权就按 `2.7` 停止。
- 浏览器验收时至少覆盖这 6 个检查点里与任务相关的部分：页面进入、关键操作、结果落点、刷新后保持、返回或切换后的再次进入、异常输入或空数据下的表现。
- 不能只打开页面看静态结果；只要任务涉及点击、输入、切换、弹层、滚动、异步加载、路由跳转、列表详情联动，就必须实操。
- 不能因为“看起来差不多”给 `已完成`；必须拿到明确正证据。
- 找不到明确正证据时，默认不输出 `已完成`；能确认缺陷就判 `未完成`，证据仍不足才判 `暂时无法判定完成`。

### 7.5 验收闭环提交规则

验收闭环分两种模式执行，不要混用：

- `带 Session 提交态`：用户已提供格式合法的 `Trae Session ID`，允许执行提交、push、带 `Commit ID` / `Trae Session ID` 的输出与回填。
- `无 Session 只验收态`：用户没有提供合法 `Trae Session ID`，或明确表示这次不提供；只做验收，不做任何 git 提交和 push。

必须满足：

- 验收结论里的 `【提示词】` 必须严格对应“当前这次实际被验收的提示词”，不能回填别的轮次内容。
- 如果当前验收的是修复轮次结果，`【提示词】` 必须填写该次修复提示词，不能填写首轮提示词，也不能把首轮提示词和修复提示词拼在一起。
- 如果一次会话里先有首轮提示词，后又生成过修复提示词，后续所有围绕修复结果的验收、提交核对和最终输出，都以最新那条有效修复提示词作为当前验收依据。
- 如果当前是 `带 Session 提交态`，且工作区里存在围绕当前提示词的改动，不管最终结论是 `已完成`、`未完成` 还是 `暂时无法判定完成`，都必须先用合法 `Trae Session ID` 提交一次，再尝试 push 到远程，然后再输出验收结论。
- `带 Session 提交态` 下，用于 push 的提交 message 必须严格等于这个合法 `Trae Session ID`。
- `带 Session 提交态` 下，如果当前分支上还没有以该合法 `Trae Session ID` 为 message 的提交，先补提交再 push。
- `带 Session 提交态` 下，必须核对“当前有效 `Trae Session ID` = `git commit` message = 最终质检结论里的 `【Trae Session ID】`”；三者任一不一致，都不能继续给最终验收结论。
- `带 Session 提交态` 下，如果已经有同 message 的提交，但工作区里还有围绕当前提示词的新改动，默认先确认是否允许 amend；用户没有明确允许 amend 时，不要跳过“提交并 push”动作。
- 如果当前是 `无 Session 只验收态`，一定不能执行 `git commit`、`git push`、`git amend`，也不要因为工作区有改动就强行补提交流程。
- `Repo URL` 必须取当前仓库远端 `origin` 对应的 GitHub 仓库地址，只输出最终 URL 本体。
- `带 Session 提交态` 下，`Commit ID` 必须填这个已 push 到远端的完整 40 位提交哈希。
- `带 Session 提交态` 下，`Trae Session ID` 必须原样回填当前上下文里可用、且通过格式校验的原始值。
- `无 Session 只验收态` 下，最终输出里不要展示 `Commit ID` 和 `Trae Session ID`。
- 如果当前是 `带 Session 提交态`，且因为远端不可达、权限不足、仓库未配置 upstream 或其他 push / remote 问题导致没法正常 push，立即命中 `2.6`；先告知用户问题并恢复正常推送，恢复前不要输出验收结果。

如果当前项目属于 Excel 管理子项目，还必须追加这组强规则：

- 先形成最终验收结论，再回填 Excel，最后才允许给用户输出验收模板。这个顺序是强指令，不允许颠倒，也不允许以“先把结论发给用户、稍后补写 Excel”的方式绕过。
- 主提示词验收时，回填目标只能是当前项目对应的主提示词那一行，不要写到别的子项目。
- 修复提示词验收时，不要预先插入空白修复行；只有确认这条修复提示词真的进入验收时，才允许在对应主提示词下面新增一行。
- 新增的修复行必须复用同一个 `子文件夹名称`、`任务类型`、`编号`，并把 `提示词类型` 写成 `修复提示词`。
- 主提示词行的 `提示词类型` 固定写 `主提示词`。
- 回填字段必须至少包含：`验收结果`、`验收Repo URL`、`验收时间`；当前被验收的提示词正文直接使用该行 `提示词` 列。
- 如果当前是 `带 Session 提交态`，还必须回填：`验收Commit ID`、`验收Trae Session ID`。
- 如果当前是 `无 Session 只验收态`，`验收Commit ID`、`验收Trae Session ID` 允许留空，不得因此阻塞验收输出。
- 如果验收结论是 `未完成` 或 `暂时无法判定完成`，还必须把最终输出里的完整 `【不满意原因】` 原样回填到 `不满意原因` 列。
- 回填值必须与最终输出完全一致：
  - `验收结果 = 【任务完成情况】去掉句号后的值`
  - `验收Repo URL = 【Repo URL】`
  - `不满意原因 = 【不满意原因】`
  - 如果当前是 `带 Session 提交态`，还要满足：`验收Commit ID = 【Commit ID】`、`验收Trae Session ID = 【Trae Session ID】`
  - 如果当前是 `无 Session 只验收态`，`验收Commit ID`、`验收Trae Session ID` 必须保持空白
- 回填命令固定使用：
  - 主提示词验收：`python3 /Users/fengyin/.codex/skills/solo-create/scripts/batch_prompt_workbook.py accept --parent "<父目录>" --folder "<子文件夹名称>" --result "<已完成|未完成|暂时无法判定完成>" --dissatisfaction "<未完成时的不满意原因，已完成可留空>" --repo-url "<Repo URL 或 无>" [--commit-id "<Commit ID>"] [--trae-session-id "<Trae Session ID>"]`
  - 修复提示词验收：`python3 /Users/fengyin/.codex/skills/solo-create/scripts/batch_prompt_workbook.py accept --parent "<父目录>" --folder "<子文件夹名称>" --prompt-type "修复提示词" --prompt "<当前修复提示词>" --result "<已完成|未完成|暂时无法判定完成>" --dissatisfaction "<未完成时的不满意原因，已完成可留空>" --repo-url "<Repo URL 或 无>" [--commit-id "<Commit ID>"] [--trae-session-id "<Trae Session ID>"]`
- 回填后必须立刻复查对应行，确认验收字段已经落盘且和当前结论一致。
- 复查不能只依赖单一路径。至少要同时满足下面两类校验里与当前场景对应的结果一致，才算“回填成功”：
  - 工作簿行级校验：直接读取 `solo-create-prompts.xlsx` 的目标行，核对 `验收结果`、`不满意原因`、`验收Repo URL`、`验收时间`，以及当前模式下应有或应空的 `验收Commit ID`、`验收Trae Session ID`。
  - 脚本视图校验：使用 `locate-project`、`pick` 或其他能够返回当前记录的脚本视图再次核对目标记录。
- 如果两类校验里任意一类失败、缺字段、写错行、读到旧值，或两类校验彼此冲突，一律按“回填失败”处理，立即命中 `2.9`。
- 只要回填失败、字段缺失、写错行，或复查发现不一致，立即按 `2.9` 停止；不要先把验收模板发给用户再补写 Excel。
- 不允许因为“肉眼看到 Excel 像是已经写上了”就继续输出；必须以完成上述复查为准。

### 7.6 过程验收触发条件

只有当用户这次消息里真的贴了 Trae 过程信息，才做过程验收。

过程验收重点方向：

- 理解 prompt 是否偏了。
- 目标是否明确。
- 推理过程是否精简。
- 输出是否正常。
- 任务规划是否合理。
- 工具使用是否合理高效。
- 操作过程是否合理。
- 有没有高危操作或越界。
- 总结是否正确。
- 有没有虚假完成。

如果用户没贴过程内容，就不要凭空补过程记录。但只要结果不满意，`过程不满意` 仍然必须写成合规结论。写法只能二选一：

- 模型行为评价。
- 更深一层的代码级 / 业务逻辑级根因。

不要把业务 Bug 直接换壳写成“实现上少了校验”“没有核对结果”“没有处理好”“没有保住状态”这类空话。

## 8. 完成判断

完成状态只允许这 3 种：

- `已完成`
- `未完成`
- `暂时无法判定完成`

判断标准优先看：

- 核心行为是不是真的存在。
- 有没有明显交互问题。
- 有没有破坏原来的阅读流或使用流。
- 如果任务和缩放、筛选、切换条目有关，顺手验证这些后续动作。

两条硬规则：

- 如果当前是 `带 Session 提交态`，只有结果满足验收要求，且对应代码已经以合法 `Trae Session ID` 为 message 提交并 push 到远程，才允许输出 `已完成`。
- 如果当前是 `无 Session 只验收态`，`已完成` 只取决于验收证据本身；不要因为没有 `Trae Session ID`、没有提交、没有 push 就阻塞完成结论。
- 只要前端任务没有经过 `@浏览器` 严格实操，或实操覆盖不到关键链路，就不能输出 `已完成`。
- 只要验收证据同时存在“像是做了”和“仍可能没做完”的冲突，默认按 `未完成` 处理，不要替实现补乐观解释。

## 9. 不满意原因

如果结果没做好，或者虽然做出来了但体验不对，就必须输出两行固定内容：

- `过程不满意：...`
- `产物不满意：...`

这两行之间只允许一个换行，不要额外插空行。

### 9.1 硬约束

- `过程不满意` 默认只写一个最核心的问题点。
- `过程不满意` 只描述这次暴露出来的当前问题，不要带历史背景、基线状态、工具路径或验收过程说明。
- `产物不满意` 只写用户能直接看到、点到、感受到的问题现象，不要夹带代码补了什么、工具怎么验的。
- 如果产物侧当前确实还没法直接判断，就写 `产物不满意：暂没法判断。`
- `过程不满意`、`产物不满意` 每行都必须以标点结尾。

### 9.2 三要素

每条不满意原因都必须同时具备：

- `触发节点`
- `实际行为`
- `业务影响`

更好的写法再补：

- `根因`
- `正确做法`

### 9.3 去雷同

必须通过这 4 个检查：

- `过程不满意` 和 `产物不满意` 不能只是在同一现象上换一种说法。
- `过程不满意` 要落在模型行为或更深一层根因。
- `产物不满意` 要落在用户可见现象和需求缺口。
- 当前项目本次输出里的两行，不能和历史同类原因只做节点词或业务名词替换。

### 9.4 写法规则

- 正文至少 30 个字。
- 默认优先写成 1 到 3 句短句。
- 不要把代码现象当根因；如果一句 `过程不满意` 可以直接替换成“某 Bug 还在”“某值没更新”“某数据没显示”，说明深度还不够。
- `过程不满意` 不要写英文文案问题；英文文案只能进 `产物不满意`。
- `过程不满意`、`产物不满意` 禁止出现 `稳定`、`不稳定`、`可靠`、`不可靠`；不要用这类抽象判断词代替现象描述。
- 不要出现工具、端口、浏览器、基线之类的背景信息。
- 不要出现跨次对比词和输出级禁词。

### 9.5 生成期强约束

下面这些不是“生成完再拦截”的补救检查，而是写 `不满意原因` 当下就必须同步满足的生成约束。落笔时就按这些约束组织句子，不要先写一版再赌后处理修掉：

- `过程不满意` 必须同时具备 `触发节点`、`实际行为`、`业务影响`。
- `产物不满意` 必须落在用户可见现象、客观证据或明确需求缺口。
- `过程不满意` 必须落在模型行为评价或更深一层根因，不能只是代码现象平移。
- `过程不满意` 不能写英文文案问题。
- `过程不满意`、`产物不满意` 不能出现 `稳定`、`不稳定`、`可靠`、`不可靠`。
- `产物不满意` 不能出现工具、端口、浏览器、基线、git、push、remote、upstream 之类背景信息。
- 两行正文都必须至少 30 个字，并以标点结尾。
- `过程不满意`、`产物不满意`、`下一轮提示词` 三者不能互相改写或骨架复用。
- 只要写出的句子命中 `稳定`、`不稳定`、`可靠`、`不可靠` 任一词，不要进入后续检查，直接判定这版 `不满意原因` 无效并立即重写。

生成时再强加 8 条批次质检约束，写的时候就必须主动避开，不要等到命中后再返工：

- 写的时候直接用中文业务现象概括证据，不要把报错堆栈、控制台原文、异常类全名、日志转储或大段报错短语写进正文。
- 一次只生成一条 `过程不满意：` 和一条 `产物不满意：`，不要边想边追加第二段同类标签。
- 起句前先换掉公共骨架，不要反复使用“先做了 A，但没有先验证 B，导致 C”这类固定句法模板。
- 写 `产物不满意` 时先锁定该任务独有的用户现象，再组织表达；不要从别的记录复制半成品后只替换业务词。
- 接口路径、函数名、文件路径、异常名只能退到辅助定位，不要让这些技术名词占据句子主干。
- 两段内容必须先分工再下笔：过程段写模型行为或根因，产物段写用户现象或需求缺口，不要靠同一现象拆成两段凑数。
- 如果当前质检问题根本不在 `不满意原因`，而在 `Trae Session ID`、`User Prompt`、重复会话或别的字段，生成阶段就直接判定“不能靠改原因修复”，不要继续硬写新原因。
- 批量改表时，先锁定“本次只允许改哪一列”，再生成该列内容；不要一边生成一边顺手改别的字段。

生成顺序也必须固定：

1. 先判断这条记录是否真的属于“可通过重写不满意原因修复”。
2. 再分别确定 `过程不满意` 的触发节点、实际行为、业务影响。
3. 再分别确定 `产物不满意` 的用户现象、证据或需求缺口。
4. 最后才组织中文句子。

不要先写句子，再靠事后检查回填这些要素。

## 10. 下一轮提示词

如果验收没过，或者用户明确表示不满意，就直接生成修复提示词。

修复提示词规则：

- 必须基于已经验出来的问题来写。
- 只写当前需要补的行为。
- 必须以 `修复` 两个字开头。
- `修复` 后面必须立刻接当前要修的问题对象或用户可感知问题。
- 不要重写最初那份完整需求。
- 不要写实现指挥、文件路径、函数名、按钮名、日志原文。
- 不要夹带跨次承接词和输出级禁词。

强制去重规则：

- 不要套用固定模板骨架后只替换业务词。
- 不要把 `过程不满意` 或 `产物不满意` 的大意压缩后直接当成 `下一轮提示词`。
- 不要把上一条 `下一轮提示词` 的句法结构原样保留，只换同义词、页面名、对象名或结果词。
- 如果发现新提示词和已有提示词共享多段连续短语，直接推倒重写。
- 修复提示词默认只保留一个主动作链，不要在一句里堆叠多个“并 / 以及 / 同时 / 后还要”。

## 11. 输出契约

### 11.1 首轮提示词场景

如果当前是在生成首轮提示词，最终只输出提示词正文，不要夹带：

- 任务类型。
- 基线状态。
- 去重说明。
- 过程说明。
- 额外总结。

### 11.2 验收或修复场景

如果当前是在后续修复提示词或验收场景，必须使用唯一固定模板。所有 key 都必须用 `【】` 包裹，每个 value 都必须单独放进一个 fenced code block。

如果当前是 `带 Session 提交态` 且命中 `2.6`，不要输出下面模板；先告知 push / remote 问题，等恢复正常推送后再继续验收。

如果当前项目属于 Excel 管理子项目，但还没有完成验收信息回填，按 `2.9` 停止；不要先输出下面模板。

这里的 `【提示词】` 还有一条强规则：

- 首轮结果验收时，填写首轮提示词。
- 修复结果验收时，填写当前这次修复提示词。

### 11.3 输出前自检

最终输出前，必须逐项自检：

- route 是否唯一且与执行过程一致。
- 当前输出是否命中 `Hard Stops`。
- 当前是否已经正确判定本次属于 `带 Session 提交态` 还是 `无 Session 只验收态`。
- 当前这次浏览器验收是否实际使用了 `@浏览器`；如果没有，是否已经按 `2.7` 停止。
- 如果当前项目属于 Excel 管理子项目，验收信息是否已经成功回填到对应行；如果没有，是否已经按 `2.9` 停止。
- 如果当前项目属于 Excel 管理子项目，是否已经完成至少两类独立回填校验，而不是只跑过一次 `accept` 或只看过一次 `locate-project` 返回值。
- 输出级内容是否含有禁词。
- `过程不满意`、`产物不满意` 是否避开了 `稳定`、`不稳定`、`可靠`、`不可靠` 这 4 个强禁词。
- 如果命中过这 4 个强禁词，是否已经把该版内容整段废弃并完成重写，而不是仅删词后继续沿用原句骨架。
- `过程不满意`、`产物不满意`、`下一轮提示词` 是否互相雷同。
- `过程不满意`、`产物不满意`、`提示词` 每行是否以标点结尾。
- 如果输出了 `下一轮提示词`，它是否以 `修复` 开头。
- 如果当前是 `带 Session 提交态`，验收输出里的 `Repo URL`、`Commit ID`、`Trae Session ID`、`提示词` 是否都对应当前这次验收。
- 如果当前是 `无 Session 只验收态`，最终输出是否已经去掉 `Commit ID` 和 `Trae Session ID`。
- 如果当前项目属于 Excel 管理子项目，Excel 里的 `验收结果`、`不满意原因`、`验收Repo URL`，以及当前模式下应有或应空的 `验收Commit ID`、`验收Trae Session ID` 是否和当前输出完全一致。
- 如果当前是修复轮次验收，`【提示词】` 是否已经切换为当前修复提示词，而不是首轮提示词。
- 如果当前已经有多条修复提示词，`【提示词】` 是否对应最新且当前有效的那一条。
- 如果当前是 `带 Session 提交态`，用户输入的 `Trae Session ID`、`git commit` message、最终输出里的 `【Trae Session ID】` 是否完全一致。
- 如果当前是 `无 Session 只验收态`，是否确实没有执行 `git commit`、`git push`、`git amend`。
- `过程不满意` 是否同时具备 `触发节点`、`实际行为`、`业务影响`。
- `过程不满意` 是否落在模型行为评价或更深一层根因，而不是代码现象平移。
- `产物不满意` 是否只写用户可见现象、证据或需求缺口。
- `过程不满意`、`产物不满意` 两行是否都至少 30 个字。
- `产物不满意` 是否误写了工具、端口、浏览器、基线、git、push、remote、upstream 等背景信息。
- 生成时是否先判断了这条记录能不能只靠重写 `不满意原因` 修复，而不是默认先写。
- 是否粘贴了异常类名、堆栈原文、控制台原文、日志转储或可识别的报错文本骨架。
- `过程不满意：`、`产物不满意：` 分段标签是否都只出现 1 次。
- 当前两段是否只是套用历史话术骨架后替换业务名词。
- 当前两段是否和批内其他记录共享连续长片段或高重复短语块。
- 如果当前是在批量改表，是否已经确认本次质检问题真的属于“只改不满意原因即可修复”。
- 如果当前是在批量改表，是否已经校验除目标列外的所有单元格保持不变。
