# 项目出题循环

把这套工作流固定成循环：先读代码，再写当前轮提示词，然后优先用 Codex 内置浏览器验收结果；如果不满意，就用自然一些的话说明原因，再继续写下一轮提示词，直到需求真正处理好。

## 工作流

先记一条总规则：

- 只要进入“启动项目并实际操作”的验收阶段，默认优先使用 Codex 内置浏览器（Browser / in-app browser / `iab`）打开本地页面并完成验证。
- 只有在内置浏览器能力当前不可用、连接失败，或者当前任务被它的能力边界客观卡住时，才允许降级到桌面浏览器或其他浏览器手段。
- 一旦发生这种降级，要在验收结果里明确说当前没有按默认浏览器路径完成，不要把降级结果伪装成标准流程。
- 如果用户明确想让我走这条路径，优先把下面这些说法当成强触发词来理解：`用 Codex Browser 打开验证`、`用内置浏览器验收`、`@浏览器 打开 localhost 验证`、`用 Codex Browser 走一遍页面`。
- 如果用户已经给了本地地址，又明确提到 `Codex Browser`、`内置浏览器` 或 `@浏览器`，默认理解成：先打开这个地址，再按用户点名的动作实际点一遍，不要退回成只读代码。
- 如果用户只是说“继续验收”，而上下文里已经明确要求过 `Codex Browser`，默认继续沿用这条路径，不要自己改走桌面浏览器。

### 1. 先建立项目上下文

从当前打开的项目开始。

- 如果这是当前项目第一次跑 `solo-create`，先用 Trae 打开当前目录；优先运行 `python3 /Users/fengyin/.codex/skills/solo-create/scripts/ensure_trae_project_open.py --project-path .`。
- 这个脚本会先看当前有没有 Trae 窗口标题已经带着当前目录名；如果已经有，就不再重复打开。
- 如果没检测到当前目录对应的 Trae 窗口，才执行打开动作。
- 这一步默认只在当前项目第一次进入 `solo-create` 流程时做，不要每一轮都重复开窗口。
- 写任何提示词前都先检查仓库。
- 优先使用 `rg --files`、`rg`、`git status --short` 和 `git diff --stat` 快速了解范围。
- 当用户让你验收某一轮结果时，先看 `git diff` 里改过的文件，再补读直接相关的代码。
- 提示词必须是当前仓库专用的，不能写成任何项目都能套用的通用需求。

把上面几条理解为硬要求：

- 写提示词前先看仓库，不要上来就凭经验写。
- 优先用 `git diff` 看这一轮真正改了什么。
- 提示词必须贴着当前项目来写，不能像通用模板。

### 1.2 先记住这一轮的基线

每次准备生成新一轮提示词前，都先记录这一轮开始时的工作区状态，后面验收时一律拿它当基线。

把这一步当成生成提示词前的必做动作，不要跳过：

- 先存基线，再输出提示词。
- 如果还没成功记录基线，就不要直接把提示词发出去。
- 用户要的是“出一轮题再回来验收”，那这一轮的起点必须先被保存下来。
- 生成提示词之前，默认先实际执行一次基线记录命令，不是只在脑子里记住这件事。
- 如果这一轮没有成功写入 `.git/solo-create-baselines/`，就不能把自己当成已经按标准流程完成了出题前准备。

- 优先运行当前仓库里的 `scripts/record_round_baseline.py` 记录基线；如果仓库里没有，再回退到 `solo-create` skill 自带的同名脚本，不要因为仓库里缺这两个文件就误判成“没有基线脚本”。
- 这个脚本会把基线写到当前仓库实际对应的 git 目录下的 `solo-create-baselines/` 里。大多数普通仓库会落在 `.git/solo-create-baselines/`，如果当前目录本身就是 git 根目录，也可以直接用 `git rev-parse --git-dir` 确认最终落点，不要只靠肉眼猜路径。
- 脚本会保存当时的 `git status --short`、`git diff --stat`、完整状态清单，以及当时已经处于脏状态的文件快照。
- 如果这一轮开始前仓库里就已经有未提交改动，默认把这些视为历史背景，不直接算进本轮结果。
- 后面用户跑完这一轮回来时，不要把整个 `git diff` 都当成本轮输出，而是只看相对于这份基线新增出来的变化。

如果仓库里和 skill 里暂时都没有这个脚本，不要当没这回事：

- 要明确告诉用户当前缺少基线记录脚本，没法做到严格的轮次隔离。
- 可以继续帮用户出提示词，但要先说明这是降级处理。
- 后面一旦用户回来验收，也要明确说明当前只能基于累计改动判断。
- 不要在这种情况下假装已经完成了标准流程，也不要省略这句说明。

这条很重要：

- 不能把“当前所有未提交改动”直接等同于“这一轮 Solo 改的东西”。
- 多轮连续迭代时，要始终按“本轮开始前”和“本轮结束后”做对比。
- 只有当旧文件在这一轮又被继续修改了，才把它重新纳入本轮验收范围。

推荐做法：

- 每轮开始前优先运行一次 `python3 scripts/record_round_baseline.py`；如果仓库里没有，就运行 skill 自带脚本。
- 每轮结束后优先运行一次 `python3 scripts/compare_round_baseline.py`；如果仓库里没有，就运行 skill 自带脚本。
- 用第二个脚本的结果确认：哪些文件是这一轮新增改动，哪些文件只是延续了上一轮的旧改动。

执行时再记住这两个判断：

- “仓库里没有 `scripts/record_round_baseline.py`”不等于“当前流程没有基线脚本可用”，要继续检查 skill 自带脚本。
- 基线是否真的写成功，以脚本实际打印出来的目录为准；不要只因为预想中的 `.git/solo-create-baselines/` 路径暂时没看到，就直接判断失败。

### 1.5 按任务类型先读对应手册

任务类型确定后，先读对应的执行手册，再开始写提示词。

对应关系如下：
- `缺陷修复`： [缺陷修复手册](references/01_Bug修复任务执行手册.md)
- `代码生成`： [代码生成手册](references/02_代码生成任务执行手册.md)
- `功能迭代`： [功能迭代手册](references/03_Feature迭代任务执行手册.md)
- `代码理解`： [代码理解手册](references/04_代码理解任务执行手册.md)
- `代码重构`： [代码重构手册](references/05_代码重构任务执行手册.md)
- `工程化`： [工程化手册](references/06_工程化任务执行手册.md)

把手册当成提示词的约束层来用：

- 遵守手册里的任务目标和写法标准。
- 最终提示词里不要出现文件路径、函数名、类名和实现指令。
- 最终提示词保持简洁。
- 如果是缺陷修复类，先在代码里确认真实问题，再翻译成用户语言。

如果任务类型是 `未归类`，可以不读手册，直接结合项目和用户意图选最贴近的一种写法。

### 2. 需要时先让用户选提示词类型

如果用户没有明确说这轮属于哪一类，先让他选一个简洁的任务类型，再生成提示词。

可选类型一般包括：
- `缺陷修复`
- `功能迭代`
- `代码生成`
- `代码理解`
- `代码重构`
- `工程化`
- `未归类`
如果用户已经把类型表达得很清楚，就不要重复追问，直接往下做。

这里额外加一条默认判断，避免把类型选偏：

- 只要需求是在现有项目里新增一块用户能独立感知的新能力、新入口或新流程，默认按 `代码生成` 处理，即使它最终也会接在现有页面或现有业务里。
- 只有当需求是在已有能力上继续补规则、补交互、补展示，或者是在原流程上往前推一点，才默认按 `功能迭代` 处理。
- 如果你发现自己写出来的是“给现有列表再加筛选、给现有按钮再补状态、给现有流程再接一步”这种，就更接近 `功能迭代`；如果写出来的是“现有系统里再长出一块新的取件、提醒、分享、预约、推荐”这种，就更接近 `代码生成`。

### 3. 生成当前轮提示词

如果是在生成首轮提示词，先做一层去重判断，不要因为项目名像、类型也一样，就顺手写出几乎同一句话：

- 先看项目目录名、任务目录名或用户给出的项目名里，是否带有可复用的编号前缀。
- 然后实际运行 `python3 /Users/fengyin/.codex/skills/solo-create/scripts/query_prompt_history.py --project-path . --task-type "<任务类型>"` 查询历史，不要只靠记忆判断。
- 这个脚本会把当前项目解析出的 `prefix`、同前缀同类型的历史提示词、当时记录的 `angle` 一起打出来；即使你是在另一个目录执行，只要前缀一致，也要拿这份历史来做去重。
- 如果查到同前缀、同类型的首轮提示词记录，默认把它们视为同一题组或同一批次，不要重复产出过于接近的首轮提示词。
- 这里的“过于接近”不只是措辞重复，也包括任务切入点基本一样、用户感知问题一样、句式只换几个词这种情况。
- 同编码前缀下，如果这次还是同一个任务类型，首轮提示词必须主动换一个角度，优先改成这个项目里同样成立、但和前面题目不重样的用户可感知需求。
- 去重时优先换 `angle`，不是只做同义改写；比如上一次写的是筛选，这一次就优先考虑搜索、排序、批量操作、信息展示、流程衔接、状态反馈这类别的角度。
- 只有在代码现实里确实只剩下一个最合理的切入点时，才允许和前题接近；这种情况下也要把描述收得更贴当前仓库，不要看起来像复制模板。
- 如果拿不准编码是否算同一组，默认只要前缀明显一致，就先按需要去重处理，宁可多换一个角度，也不要重复出题。
- 如果历史脚本返回的 `matched_count` 大于 0，就先参考这些历史记录再出题；如果 `matched_count` 是 0，才按正常首轮逻辑自由出题。

按最适合当前任务的角度去写提示词：

- 如果是新的需求，就按选定类型写一版贴着仓库现状的提示词。
- 如果上一轮只做成了一部分，或者做偏了，下一轮默认从修复问题的角度去写，除非用户另外指定。
- 如果上一轮已经暴露出明确问题，下一轮就围绕这些具体问题写，不要把最初的大需求整段重写一遍。
- 如果需求是在当前系统里新增一块完整能力，哪怕要复用现有页面、现有数据或现有接口语境，也默认先按 `代码生成` 的写法出题，不要顺手写成 `功能迭代`。

再补一条出题偏好，专门约束 `代码生成` 和 `功能迭代`：

- 这两类提示词默认优先挑那种会自然牵动多个代码文件的需求角度，不要总出只改一个页面局部文案、一个小开关或一个孤立筛选条件就能做完的题。
- 更优先选择会同时带动页面、状态数据、路由、列表详情联动、统计区、弹窗、表单、导航入口、空状态、结果反馈这类两个及以上层面的需求。
- 这里说的“多文件修改”要通过用户可感知的完整行为来带出来，不要在提示词里明说“需要改多个文件”。
- 如果仓库现实里既有“单点小修”也有“跨页面或跨模块联动”的切入点，默认优先选后者。
- 只有当当前项目的真实结构决定了最合理的需求本来就只能落在单文件时，才允许继续出单文件倾向的题；不要为了凑多文件硬拗一个不自然的需求。
- 判断一个角度是否更合适时，优先看它会不会自然涉及至少两层：数据来源或状态、页面展示、交互流程、入口衔接、结果反馈。
- 如果是 `功能迭代`，默认优先出“现有能力再往前走一段”的题，比如筛选结果联动统计、列表联动详情、状态变化联动卡片或概览，而不是只补一个孤立控件。
- 如果是 `代码生成`，默认优先出“在现有系统里长出一块完整新能力”的题，这块能力最好天然带入口、主体页面或弹窗、数据承接和结果展示，不要退化成只多一个按钮或只多一张静态卡片。

提示词写法要求：

- 语气要像真实同事或产品在和开发沟通。
- 内容要贴着已经观察到的行为和预期结果来写。
- 优先写用户能直接感知到的现象，不要上来就给实现提示。
- 默认用中文，除非用户明确要求别的语言。
- 避免机器味很重的模板话术，也不要写成验收报告腔。
- 不要解释过多，通常一小段就够了。
- 如果当前类型有对应手册，要遵守它的硬约束。
- 如果当前类型是 `缺陷修复`，提示词默认用“修复……问题”的句式开头，直接点出要修的现象。
- 默认不要在提示词里用引号包住页面文案、功能名或问题现象，尽量改写成自然转述。

再额外收紧几条生成偏好：

- 默认优先写得像人会顺手发给同事的一句话，不要写成刻意完整的说明文。
- 能用更自然的口语化动词时，优先不用太书面的抽象词，比如更偏向“改一点”“反复检查”“维护费劲”，少用“调整细节”“反复核对”“维护成本高”。
- 优先让句子里带一点具体动作感，读起来像真遇到过这个麻烦，而不是只在概括问题类型。
- 如果两种表达都成立，默认选更短、更顺口、更像维护者抱怨的那种。
- 除非有必要，不要在一句话里同时塞“问题背景 + 目标 + 约束 + 总结评价”四层意思，够说明白就停。

如果当前类型是 `代码重构`，默认再遵守这些偏好：

- 优先从“代码越来越难改”“逻辑缠在一起”“改一点要来回看很多地方”这类维护者体感切入，不要写成空泛的技术治理口号。
- 描述问题时，默认更偏向“耦合得太紧”“牵一处动很多处”“维护起来费劲”这种自然说法。
- 写目标时，优先用“让后面维护别这么费劲”“后续改起来轻一点”“别一改就到处翻”这类结果表达，不要上来就写“提升可维护性”“优化代码结构”。
- 在“不影响现有功能”的约束之外，默认不要再叠加多余的正确话术，避免把一句提示词写得太满。

生成提示词时，默认连带做到这件事：

- 先确认这一轮基线已经成功记录。
- 如果这是首轮提示词，再先查一次同前缀同类型历史，确认这一版没有撞上已经用过的 `angle`。
- 然后再把提示词发给用户。
- 提示词发出去以后，立刻运行 `python3 /Users/fengyin/.codex/skills/solo-create/scripts/record_prompt_history.py --project-path . --task-type "<任务类型>" --angle "<这次的切入点>" --prompt "<最终提示词>"` 记一条历史，确保下次在别的目录也能查到。
- 提示词发完后，下一次只要用户表示“生成完了”“继续”“验收”“看结果”，默认都进入验收流程，不要再停留在讨论态。
- “确认基线已经成功记录”指的是：要么脚本执行成功，要么明确告诉用户当前只能降级处理；不能什么都不做就直接进入下一步。
- “记录提示词历史”也算标准流程的一部分；如果这一步没成功，要明确说当前首轮去重只能靠当前上下文，不能假装已经完成跨目录去重。

如果是验收之后的下一轮提示词，额外遵守这几条：

- 要把它写成修复已确认问题的后续需求，而不是把整个功能从头说一遍。
- 直接点出能看见的问题，比如自动跳转、按钮一直禁用、图表没对齐、高亮偏移。
- 只有约束确实存在而且有用时才写进去。
- 如果这一轮本质上是在补问题，即使最初任务不是 `缺陷修复`，下一轮提示词也默认切到“修复……问题”的表达。
- 不要出现“收一下”“没收干净”“再收一收”“这块还没收住”这类空泛口语。
- 默认不要在下一轮提示词里出现引号；就算要提到页面上的一块区域或某个现象，也改成自然描述，不要直接把页面文案用引号框起来。

## 验收循环

用户跑完一轮、仓库里已经有改动后，要自己去验结果，不要只看代码表面。

只要用户是在上一轮提示词之后回来，并且表达的是“继续”“看下结果”“验收一下”“生成完了”这类意思，默认动作不是再聊需求，而是立刻进入下面这套验收循环。

把这句当成强制切换规则：

- 用户一旦是在上一轮提示词之后回来让继续，就不要再给普通解释型回复。
- 默认立即做这四步：读相对基线改动、启动项目、实际操作、输出固定格式结论。
- 如果其中任何一步没做，就不能算按标准流程完成了这次验收。

### 1. 先读改动代码

- 先拿当前状态和本轮开始前记录的基线做对比。
- 再看 `git status --short` 和 `git diff --stat` 里相对于基线新增的变化。
- 读完这轮真正新增或继续修改过的文件。
- 结合上一轮提示词和这次改动，判断目标行为本来应该是什么。

如果已经记录过基线，优先运行仓库里的 `scripts/compare_round_baseline.py`；如果仓库里没有，就运行 skill 自带脚本，按它输出的“本轮新增文件”和“本轮继续修改文件”来确定验收范围。

如果发现下面这种情况，要主动区分：

- 某些文件在上一轮开始前就已经改过，这一轮没有继续动它们。
- 某些文件是旧改动文件，但这一轮又被继续修改了。

处理规则：

- 第一种不要直接算进本轮验收。
- 第二种可以纳入本轮，但要只关注这一轮新加上去的变化和它带来的结果。

如果当前线程里没有记录过本轮基线，没法可靠区分新旧改动，就不要硬判：

- 先补记一份新的基线，作为下一轮的起点。
- 如果用户现在就要你判断这一轮结果，就先明确说明：当前只能基于累计改动来判断，没法百分百分离这一轮和前几轮的边界。
- 这句说明必须出现在验收结果里，不要只在心里知道。

### 2. 启动项目并实际操作

- 用仓库真实的启动命令把项目跑起来。
- 需要时可以先构建，但不能把构建通过当成任务完成的证据。
- 如果是前端或浏览器可见的任务，要自己打开页面走一遍改动后的流程。
- 如果用户要求本地页面验收，就必须优先用 Codex 内置浏览器去操作，不要停在读代码这一步。
- 如果用户已经明确说“用 Codex Browser 打开验证”或同类说法，默认先检查当前线程里有没有 in-app browser 上下文；如果已经有当前页地址，就直接接着这页验，不要先绕去别的浏览器。
- 如果用户给了地址但没有写具体动作，默认至少完成这三步：打开页面、登录或进入目标区域、验证这轮改动对应的核心行为。
- 需要打开浏览器页面时，默认在内置浏览器里新建标签页做验收；只有用户明确要求复用当前页时，才允许这么做。
- 如果内置浏览器当前不可用，先明确说明阻塞，再决定是否降级到桌面浏览器继续验收。
- 如果内置浏览器当前不可用、连接失败，或者打开后无法稳定停留在目标页面，默认先看是否还能通过代码变更判断结果；只要代码改动已经足够支撑判断，就直接基于代码判断是否完成，不必强行继续切桌面浏览器。

把这一步理解成默认强制项：

- 用户让继续后，优先做启动和实际操作，不要只做代码审阅。
- 只要项目理论上可以运行，就要先尝试真实启动。
- 只有在本机缺少依赖、端口冲突、服务起不来、界面工具不可用这类客观阻塞下，才允许停在“未完成验收”。
- 如果页面侧验收被内置浏览器问题卡住，但代码改动已经能明确说明需求是否落地，优先给出基于代码变更的完成判断；只有代码和页面两边都没法支撑判断时，才落到“暂时无法判定完成”。
- 如果启动时遇到端口冲突，默认不要去停掉当前已经运行的服务；优先从原端口 `+1` 开始往后找可用端口，找到后就在新端口上继续启动，并记住后续验收都要跟着这个新端口走。
- 如果卡住了，要把阻塞原因说清楚，但不能因为有阻塞就跳过“是否完成”的判断逻辑。
- 不要因为已经看过代码，或已经看到用户说“跑完了”，就省略真实启动和操作。
- 浏览器验收时，优先保护用户当前正在看的页面；默认通过内置浏览器的新标签页进入目标地址，不要去改写用户桌面浏览器当前活跃页。
- 只要内置浏览器可用，就不要再优先选 Chrome、Safari 或桌面自动化去做本地验收。
- 如果当前消息已经出现 `# In app browser` 和 `Current URL` 这类上下文，默认说明内置浏览器页面已经在当前线程里打开了；这种情况下，优先基于这页继续验，不要再说“拿不到浏览器页面”。

### 3. 判断任务是不是真的完成

判断标准是“用户要的行为有没有真的实现”，不是“代码看起来像是写了”。

至少检查这些点：
- 主要要求的行为是不是真的存在。
- 新结果有没有破坏原来的阅读流或使用流。
- 还有没有明显的对齐、跳转、禁用态、加载态或交互问题。
- 如果这个任务和缩放、调整尺寸、筛选、切换条目有关，就顺手验证这些后续动作还能不能正常工作。

输出结论时要收紧成这三种之一：

- 已完成：真实运行和操作后，核心行为符合要求，也没有明显交互问题。
- 已完成：真实运行和操作后，核心行为符合要求，也没有明显交互问题；或者内置浏览器不可用，但代码变更已经足够明确说明需求已落地。
- 未完成：已经实际验过，确认有行为偏差、体验问题或结果不对；或者内置浏览器不可用，但代码变更已经足够明确说明需求还没做对。
- 暂时无法判定完成：被运行环境或启动阻塞卡住，而且代码变更也不足以支撑完成判断。

其中第二种和第三种都不能只停在结论上，必须继续给出不满意原因和下一轮提示词。

## 输出不满意原因

如果结果没做好，或者虽然做出来了但体验不对，就要输出不满意原因，而且语气要像真人评审。

写法规则：

- 不满意原因正文至少 `30` 个字。
- 要像真人在评审这次改动，不要像机器在写审计报告。
- 先说具体问题，不要先贴总结标签。
- 只写问题本身，不写评价口吻、流程说明、分析结论或客套铺垫。
- 只允许描述这轮提示词本身对应的完成情况和用户可感知的问题，不要把基线记录、浏览器选型、验收流程、端口处理、工具降级这类流程信息塞进不满意原因。
- 把页面上发生了什么、操作时出了什么问题写清楚。
- 用词直接一点，尽量像日常说话。
- 默认不要在不满意原因里用引号包住页面文案、区域名或问题现象，优先改写成自然描述。
- 默认不要出现 `AI`、`人工智能`、`机器味`、`模型`、`像机器人写的` 这类词，也不要出现“体验未达预期”“整体还不行”“功能基本有了”这种空话。
- 避免“总体来看”“关键体验未达到预期”“能力已具备但”这种报告腔收尾。
- 除非用户明确要列表，否则不要拆成很机械的条目。
- 除非用户明确要代码原因，否则不要在面向用户的不满意原因里脑补技术根因。
- 默认只描述用户能直接感知到的现象，不展开实现层分析。
- 尽量不要引用页面里的具体文案、按钮名、字段名、标签名、报错原文或测试数据编号，除非不写出来就说不清问题。
- 尽量不要堆具体数字、页码、章节号、坐标、百分比之类的细节；只有当这些数字正是问题本身的一部分时才保留。
- 不要把不满意原因写成“做了什么功能 + 哪些点待优化”的工作汇报，重点始终放在现在实际哪里不好用。
- 不要在不满意原因里写“没达到要求”“不符合预期”“还不能算完成”这类判断句，直接改成哪里点不动、哪里没显示、哪里会跳错、哪里前后对不上。
- 如果某个问题不引用页面原文也能讲清楚，就不要把页面上的字样原封不动写出来，更不要额外套引号。
- 默认不要先交代“已经做了什么”或“哪部分已经有了”，直接从不好用的地方说起。
- 只有在不提已完成部分就说不清问题边界时，才允许用半句话轻带一下，而且不要把它写成表扬或进度总结。
- 默认优先写成 2 到 3 句短句，不追求把背景、影响、原因一次性说全。
- 能用“这里还是有问题”“这个地方还不对”“有时候会乱”这种更口语的说法讲清楚，就不要硬写成长句分析。
- 只描述已经看到的问题现象，不要写泛化判断，不要写“这个功能对用户来说还是悬着的”“现在还不能放心用”这类空话。
- 不要把“没验收完”“没跑起来”扩写成价值判断，直接说卡在哪、哪一步没过、哪里还不对就行。
- 不要写“收一下”“没收干净”“还差最后收尾”这类模糊表达。
- 不要用“这块没收住”“细节还没收好”代替具体问题。
- 不要写“稳定”“更稳定”“稳定一点”“稳定可用”这类词，哪怕是在不满意原因里，也必须改成具体表现。
- 不要写“串上”“接住”“连上”“链路”“串起来”“串掉”“怀疑”这类词，必须改成更具体的用户可感知说法，比如“进到这里”“跟前一步对上”“整段流程能走完”“看起来像填错了账号”。
- 默认不要在不满意原因里出现“顺畅”或“没落下来”这类空泛收尾词，直接改成具体哪里卡住、哪里没显示出来、哪里前后对不上。

推荐结构：

用 2 到 4 句连贯的话带出这几层意思：
- 还有什么地方表现不对。
- 为什么这和原本要求对不上。
- 实际用起来还卡在哪，为什么不方便。

如果其中某一层不说也能成立，就可以省掉，不要为了凑完整结构把话写重。

目标语气示例：

搜索结果分组和侧边栏是有了，但实际用下来，搜索一完成就会直接跳到第一个结果，把当前阅读位置打断了，这点和“不影响当前阅读”的要求对不上。另外高亮效果也不太对。现在不是贴着命中文字去高亮，而是会出现一大片偏移的遮罩，视觉上比较乱，定位内容也不方便。

把这种语气当成标准，但不要机械照抄，要根据当时的问题来写。

如果想让表达更稳一些，可以额外遵守这几条：

- 先用更抽象的用户感知来概括问题，比如“跳转位置不准”“阅读流被打断”“高亮范围不贴合内容”，再决定是否需要补一个具体例子。
- 如果一个问题可以不靠页面文案也讲清楚，就不要引用文案本身。
- 如果一段话里出现太多引号、括号、数字或专有名词，通常已经写得太贴代码和页面了，要再收一层。
- 如果开头一句是在介绍这轮“做到了什么”，通常已经偏了，优先改成“现在哪里还是不对”。
- 如果一句话已经把问题讲清楚了，不要再补一整句解释它为什么算问题。
- 优先写“现象 + 不对劲的点”，少写“现象 + 推断 + 评价 + 总结”。

反例提醒：

- 不要写“这个功能对用户来说还是悬着的”。
- 不要写“整体上还不能算稳定可用”。
- 不要写“目前只能算初步完成”。
- 不要写“还没串上”“没有接住”“没有连上”“整条流程没走通”“内容串起来了”“输入框串掉了”“让人怀疑是不是填错了”这类话。
- 默认也不要写“接住”“连上”这两个词，就算单独出现也不行，改成“看到”“同步过来”“走到下一步”“显示正常”这种更具体的说法。
- 不要写“因为基线没记好”“这次没走默认浏览器”“端口又占了”“工具降级了，所以先这样”这类流程说明。
- 这些都不是问题本身，要改成具体描述哪里没跑通、哪里会中断、哪里显示不对。
- 不要写带引号的页面原文转述，比如把某块区域名或按钮文案直接用引号包起来重复一遍。

只要验收结论不是“已完成”，默认都要输出这一段不满意原因；不要等用户再追问一次。

额外收紧一条：

- 不满意原因默认只写 1 到 3 句。
- 句子可以短，但合起来默认不少于 `30` 个字。
- 够说明问题就停，不要展开成长分析。
- 如果是“暂时无法判定完成”，不满意原因就直接说阻塞点，不额外铺陈。

## 生成下一轮提示词

如果验收没过，或者用户明确表示不满意，就直接生成下一轮提示词。

下一轮提示词规则：

- 必须基于已经验出来的问题来写，不能靠猜。
- 把它当成一次后续修复需求。
- 比最初那版提示词写得更收敛一些。
- 保留用户偏好的语气：自然、直接、少模板、少机器味。
- 如果上一轮大部分都已经做到了，只剩几个问题，就只盯着这些问题写。
- 默认继续沿用最开始生成提示词时的约束习惯：少实现味、少页面文案、少测试痕迹，更像真实需求补充。
- 默认不要出现文件路径、函数名、类名、接口名、按钮名、字段名、报错原文或测试脚本里的专有叫法。
- 默认不要在下一轮提示词里写带引号的页面文案或问题描述。
- 默认不要把验收时看到的具体页码、章节号、百分比、坐标、日志编号直接抄进提示词里，除非这些细节本身就是需求核心。
- 优先把问题翻译成用户视角下的行为偏差，而不是复述一次验收过程。
- 直接描述需要补上的行为，不要写“先看一下”“收一收”“查一查”“搜一搜”“保证一下”这类带指挥口气的铺垫。
- 提示词本体尽量一步到位，说清要做到什么，不要用任务过程型措辞。
- 不要写“再顺一下”“优化一下”“补一下”“调整一下”这种空泛过渡，直接说要改成什么结果。
- 默认不要用“把这个功能”“把这一块”“把这部分”这种先概括再展开的起手式，直接进入具体需求。
- 默认不要写“稳定”“更稳定”“稳定一点”“稳定查看”“稳定使用”“稳定展示”这类空泛结果词，优先改成具体可见的行为结果，比如“结果别乱”“切换后内容别丢”“分页要接着当前筛选结果走”。
- 默认不要写“串上”“接住”“连上”“链路”“串起来”“串掉”“怀疑”这类抽象词，优先改成“能在这里看到前面生成的记录”“点完后能继续往下走”“从这里进去后别跳丢”“账号显示要正常”这种更能直接验的说法。
- 默认不要在下一轮提示词里出现“顺畅”或“没落下来”，改成更具体的页面结果，比如“能直接看到”“更新后会同步过来”“切过去别丢内容”。
- 如果是修复类提示词，默认写成“修复……问题……”，不要写成“让……”“支持……”“把……改成……”，除非用户明确要求别的口吻。
- 不要写“收一下”“收干净”“没收干净”“把尾巴收掉”这类没有可验结果的表达。

更合适的写法方向：

- “修复搜索完成后自动跳到第一个结果的问题，保留当前阅读位置……”
- “修复高亮范围偏移和缩放后错位的问题……”
- “让推荐路线在现有导航流程里能直接生成完整结果，并把每站看点和时间安排展示清楚……”
- “分类切换要能完整显示，导览卡片整块都能点击，点选后地图和详情同步更新……”
- “修复筛选后结果变乱和分页对不上的问题，翻页后也要接着当前筛选结果显示……”

如果类型是 `缺陷修复`，把下面这种格式当成默认模板：

- “修复……问题，……”
- “修复……和……问题，……”

例如：

- “修复搜索完成后自动跳转的问题，保留当前阅读位置。”
- “修复高亮范围偏移和缩放后错位的问题。”

避免这样做：

- 不要把最初那份完整需求整段重写。
- 不要顺手加无关增强项。
- 不要写成看不出具体问题的空泛验收话术。
- 不要把页面文案、按钮文字、日志内容或测试样例直接拼进提示词里。
- 不要写成“修复 A、修复 B、修复 C”的任务单口吻，尽量仍然保持一句自然的需求表达。
- 不要把提示词写成对开发的实现指挥，也不要写成对上一轮产出的复盘说明。
- 不要写“先把这个功能收一收”“先处理一下”“先排查一下”这种过程导向的话。
- 不要写“搜一搜”“看一眼”“顺手把”这类口语化动作词。
- 不要写“再顺一下”“优化下这个功能”“把这块再完善一下”这类没有明确结果的描述。
- 不要先说“功能”本身，再说要做什么，直接写要补的行为和结果。
- 不要在修复类提示词里写成需求新增口吻；能写“修复……问题”的，就不要改写成“增加……能力”“支持……效果”。
- 不要写“稳定怎么怎么样”“更稳定一点”“先把这个做稳定”这类泛化说法，必须落到能验的页面行为上。
- 不要写“串上”“接住”“连上”“链路”“串起来”“串掉”“怀疑”这类词，必须换成页面里能直接看到、点到、走到的结果描述。

把这条改成默认联动输出：

- 只要这一轮验收没过，回复里就同时给两样东西：不满意原因 + 下一轮提示词。
- 不要只说问题，不补下一轮提示词。
- 不要只给下一轮提示词，不解释这次为什么没过。
- 如果是“暂时无法判定完成”，也要先说清楚卡点，再补一版围绕当前已确认问题或阻塞点的下一轮提示词。

再补一条更硬的默认输出：

- 在“上一轮提示词已经跑完，用户回来让继续”的场景里，默认最终只给三项核心结果：`是否完成`、`不满意原因`、`下一轮提示词`。
- 如果已完成，只保留 `是否完成：已完成`，不要再补总结。
- 如果未完成或暂时无法判定完成，就按这三项固定顺序输出，不额外加分析、过程复述、代码审阅总结。
- 不要把这三项拆开到多条回复里。
- 不要把“我先去看一下”“我已经帮你验了”这种过程说明混进最终结果里。

## 输出形式

根据用户当前的要求决定输出形式：

- 如果用户要提示词，就直接给提示词。
- 如果用户问这轮是不是完成了，就给结论和简短理由。
- 如果用户要不满意原因，就只输出自然语言版本的不满意原因。
- 如果用户要下一轮提示词，默认从修复问题的角度写，除非上下文明显属于别的类型。

但在“上一轮提示词已经跑完，用户回来让继续”的场景下，默认输出顺序固定为：

- 先给验收结论。
- 如果没过或没法完成验收，再给不满意原因。
- 然后直接给下一轮提示词。

默认格式尽量简洁，直接按下面这样给：

- `是否完成：已完成`

或

- `是否完成：未完成`
- `不满意原因：...`
- `下一轮提示词：...`

或

- `是否完成：暂时无法判定完成`
- `不满意原因：...`
- `下一轮提示词：...`

把这个格式当成默认模板直接套用：

- 用户在上一轮提示词之后说“继续”，最终回复默认就按这三行结构收口。
- 即使当前因为缺基线脚本或启动失败而降级，也还是按这三行结构输出。
- 除非用户明确只要其中一项，否则不要改成自由发挥的段落总结。

## 默认处理方式

- 默认当前工作区里的仓库就是目标项目。
- 只要任务是用户可见的，就默认验收要覆盖真实运行行为。
- 默认“相对于本轮基线新增的改动文件”才是一轮验收时最重要的检查面。
- 只要用户没有明确停下或切任务，就继续这套循环，直到需求真正处理好。
- 基线文件默认只存放在当前仓库实际 git 目录下的 `solo-create-baselines/` 里，普通仓库通常表现为 `.git/solo-create-baselines/`，作为本地辅助记录，不当成业务产物，也不需要提交。

再补两条硬规则：

- 每一轮“出提示词”之前，先做基线记录；每一轮“继续”之后，先做真实启动验收。
- 验收失败时，默认一次性把“不满意原因”和“下一轮提示词”都给出来，不要拆成两轮对话。
- 验收输出默认只保留核心结果，不主动附加“我做了什么”“我看了哪些文件”“我接下来建议什么”这类补充，除非用户明确要看。
- 如果这一轮没有成功执行基线记录，就要明确标记成“未按标准流程完成基线记录”，不能假装这一步已经完成。
- 如果用户说“继续”之后没有走真实启动验收，就要把它视为流程执行错误，下一次必须纠正，不要重复同样的跳步。

## 多项目并行隔离规则

如果用户在不同项目里同时跑多轮任务，默认允许并行，但必须把它们当成彼此独立的验收单元来处理。

先遵守这几个基本原则：

- 一个项目只对应一个工作目录，不混用仓库。
- 一个项目只对应一组端口，启动前先确认不会和别的项目撞端口。
- 一个项目只对应自己的基线记录，只在当前项目实际 git 目录下的 `solo-create-baselines/` 里读写；普通仓库通常表现为 `.git/solo-create-baselines/`。
- 一个项目只对应内置浏览器里的一组固定标签页，验收到结束前不要和别的项目混着用。
- 如果需要在浏览器里打开项目页面，默认在内置浏览器里为对应项目新建标签页，不要顶掉用户当前正在看的其他页面。

并行时的操作规则：

- 先分别启动各自项目服务，再逐个确认访问地址和端口。
- 如果两个项目默认端口相同，默认从原端口 `+1` 开始往后找空闲端口，不要通过停掉已经在跑的那个服务来抢端口。
- 内置浏览器里要明确记住“哪个标签页属于哪个项目”，不要在多个项目之间随意复用同一个标签页。
- 打开新页面时，默认在目标项目自己的内置浏览器标签页里继续，不要直接改写别的项目当前页。
- 如果任务涉及登录态、cookie、`localStorage` 或其他浏览器本地状态，默认认为它们可能互相影响；同源项目尽量拆到不同标签页上下文，必要时重新开新标签重新登录。
- 如果任务需要我实际操作浏览器界面，默认按串行验收处理，不在多个项目标签页之间高频来回切换。

判断是否可能互相影响时，重点看这几类共享资源：

- 端口是否重复。
- 内置浏览器状态是否共用。
- 是否连着同一个本地数据库、后端服务、测试账号或环境变量。
- 是否需要我同时操作多个浏览器标签页或降级后的桌面窗口。

如果发现下面任一情况，不要直接并行验收：

- 两个项目使用同一个端口，且暂时没改开。
- 两个项目依赖同一个登录态或同源浏览器存储。
- 两个项目实际上连着同一套会互相污染的数据环境。
- 当前没法可靠区分哪个内置浏览器标签页属于哪个项目。

遇到这些情况时，优先这样处理：

- 先把端口拆开；如果冲突了，默认按 `原端口 + 1`、`+2` 这样往后找空闲端口，不要先去停现有服务。
- 先把内置浏览器标签页拆开；如果不得不降级，再把桌面浏览器窗口拆开。
- 先把项目逐个验收，而不是同时点多个标签页或多个窗口。
- 如果仍然存在共享状态，就明确告诉用户：当前可以并行启动，但不适合并行操作和并行验收。
